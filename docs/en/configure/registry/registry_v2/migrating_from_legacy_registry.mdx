# Registry v2: Migrating from Legacy Registry

Migration from the legacy ACP Registry is required when an existing ACP environment moves image workloads to **Registry v2** in `image-registry-system`.
Registry v2 means the Registry deployed and managed by `cluster-image-registry-operator`.

Use this page to plan the migration path, perform the required Registry v2 configuration, copy image data when needed, backfill Image API metadata, and verify the cutover.
For large batches or repeatable migration automation, use the validated Registry v2 migration runbook in the ACP knowledge base.

During migration, use a maintenance window when possible.
At minimum, pause writes to the legacy Registry.
Reads can remain available.
If new images are pushed during migration, repeat the copy and metadata backfill steps before the final cutover.

## Migration Scenarios

| Scenario | Use when | Blob migration | Metadata migration |
| --- | --- | --- | --- |
| Storage reuse | The legacy Registry and Registry v2 run in the same ACP environment and business cluster, and Registry v2 can read the same blob storage. | Not required. Configure Registry v2 to use the existing blob storage. | Required. Backfill `ImageStream` and `ImageStreamMapping` objects for Registry v2. |
| Cross-registry sync | The legacy Registry and Registry v2 use different storage, clusters, or ACP environments. | Required. Copy images from the legacy Registry to Registry v2. | Required after image copy. Backfill `ImageStream` and `ImageStreamMapping` objects for Registry v2. |

## Prerequisites

- Registry v2 is installed and available in `image-registry-system`.
- You have administrator access to the source cluster and target cluster.
- You can log in to ACP with `ac` and select the target business cluster.
- You can access the legacy Registry and Registry v2 endpoints from the migration workstation.
- For storage reuse, you know the legacy storage backend, credentials, path prefix, and retention requirements.
- For cross-registry sync, Registry v2 has enough storage capacity for the copied images.
- You have a maintenance window or a confirmed plan to pause writes to the legacy Registry.

## Prepare Migration Inputs

Record the source and target Registry endpoints:

```bash
export LEGACY_REGISTRY_URL=http://old-registry.example.com:5000
export LEGACY_REGISTRY_HOST=old-registry.example.com:5000
export TARGET_REGISTRY_HOST=image-registry.example.com
export MIGRATION_DIR=/tmp/registry-migration
mkdir -p "$MIGRATION_DIR"
```

Log in to ACP and select the target cluster:

```bash
ac login https://acp.example.com --name registry-migration
ac config get-clusters
ac config use-cluster <cluster-name>
```

Check the current Registry mode:

```bash
ac config get-registry-mode
ac registry info
```

If the context can access both legacy Registry and Registry v2, select the backend explicitly during migration:

```bash
ac config set-registry-mode legacy
ac registry info
```

Expected results:

- `ac registry info` returns the legacy Registry endpoint while the mode is `legacy`.
- You can reach `LEGACY_REGISTRY_URL` and `TARGET_REGISTRY_HOST` from the migration workstation.

## Prepare the Image List

Export the legacy image list visible to the migration account:

```bash
ac config set-registry-mode legacy
ac get images --registry-url "$LEGACY_REGISTRY_URL" > "$MIGRATION_DIR/legacy-images.txt"
```

Review the output and create a migration list that contains only the images to migrate. Each line must use the `namespace/name:tag` format:

```text
team-a/my-app:v1
team-a/my-api:v2
```

Save the list as:

```bash
export IMAGE_LIST_FILE="$MIGRATION_DIR/image-list.txt"
```

Verify the migration scope:

```bash
sed -n '1,20p' "$IMAGE_LIST_FILE"
wc -l "$IMAGE_LIST_FILE"
```

Remove temporary test images, retired namespaces, and incorrect tags before continuing.

## Write Registry Credentials

Write credentials for both Registry endpoints into the same OCI auth file:

```bash
export REGISTRY_AUTH_DIR="$MIGRATION_DIR/registry-auth"
export REGISTRY_AUTH_FILE="$REGISTRY_AUTH_DIR/config.json"
mkdir -p "$REGISTRY_AUTH_DIR"
export DOCKER_CONFIG="$REGISTRY_AUTH_DIR"
```

```bash
ac registry login \
  --registry "$LEGACY_REGISTRY_HOST" \
  --skip-check \
  --to "$REGISTRY_AUTH_FILE"

ac registry login \
  --registry "$TARGET_REGISTRY_HOST" \
  --skip-check \
  --to "$REGISTRY_AUTH_FILE"
```

Use `--insecure` only when the Registry endpoint uses HTTP or a certificate that the migration workstation does not trust.

## Scenario A: Reuse Legacy Storage

Use this path only when Registry v2 can safely read the same blob storage as the legacy Registry.

Before changing Registry v2 storage, confirm the following:

- Writes to the legacy Registry are paused.
- The legacy storage backend supports being attached to Registry v2 with the same bucket, path prefix, filesystem path, or PVC data.
- The credentials, TLS settings, and custom CA configuration are available for Registry v2.
- The storage administrator has approved the reuse plan.

Configure Registry v2 storage by using the appropriate `Config/cluster.spec.storage` settings. For examples, see [Setting up and configuring the registry](./setting_up_and_configuring_the_registry.mdx).

Apply the storage configuration:

```bash
kubectl apply -f registry-v2-storage.yaml
```

Verify that Registry v2 picked up the storage configuration:

```bash
kubectl get configs.imageregistry.operator.alauda.io cluster -o yaml
kubectl -n image-registry-system rollout status deployment/image-registry --timeout=300s
kubectl -n image-registry-system rollout status deployment/image-api-server --timeout=300s
```

Expected results:

- `Config/cluster` reports `Available=True`, `Progressing=False`, and `Degraded=False`.
- `image-registry` and `image-api-server` roll out successfully.

Do not run image pruning or Registry garbage collection against either Registry until migration is accepted.

## Scenario B: Copy Images Between Registries

Use this path when Registry v2 cannot reuse the legacy Registry storage.

Create a mirror mapping file that maps each source image to the target Registry. Each line uses the `source=target` format:

```text
old-registry.example.com:5000/team-a/my-app:v1=image-registry.example.com/team-a/my-app:v1
old-registry.example.com:5000/team-a/my-api:v2=image-registry.example.com/team-a/my-api:v2
```

Save the file as:

```bash
export MAPPING_FILE="$MIGRATION_DIR/mirror-map.txt"
```

Run a dry run:

```bash
ac image mirror \
  -f "$MAPPING_FILE" \
  --dry-run \
  --keep-manifest-list
```

After reviewing the plan, copy the images:

```bash
ac image mirror \
  -f "$MAPPING_FILE" \
  --continue-on-error \
  --keep-manifest-list \
  --max-per-registry 4
```

Expected results:

- The dry run shows the expected source and target references.
- The copy command completes without unexplained authorization, storage, or network failures.
- Multi-architecture images retain their manifest lists.

## Backfill Image API Metadata

Registry v2 needs Image API metadata in addition to image blobs. Backfill metadata by creating `ImageStream` and `ImageStreamMapping` resources for migrated images.

For each migrated image, inspect the digest and media type from the Registry that contains the blobs:

```bash
ac image info "$TARGET_REGISTRY_HOST/team-a/my-app:v1"
```

For storage reuse, inspect the legacy Registry if Registry v2 cannot resolve the tag before metadata backfill:

```bash
ac image info "$LEGACY_REGISTRY_HOST/team-a/my-app:v1"
```

Create an `ImageStream` manifest named `team-a-my-app-imagestream.yaml`:

```yaml
apiVersion: image.alauda.io/v1
kind: ImageStream
metadata:
  name: my-app
  namespace: team-a
spec:
  lookupPolicy:
    local: false
```

Apply the `ImageStream`:

```bash
kubectl apply -f team-a-my-app-imagestream.yaml
```

Create an `ImageStreamMapping` manifest named `team-a-my-app-v1-mapping.yaml`:

```yaml
apiVersion: image.alauda.io/v1
kind: ImageStreamMapping
metadata:
  name: my-app
  namespace: team-a
image:
  metadata:
    name: "sha256:<digest>"
  dockerImageReference: "image-registry.example.com/team-a/my-app@sha256:<digest>"
  dockerImageManifestMediaType: "application/vnd.docker.distribution.manifest.v2+json"
tag: "v1"
```

Create the mapping:

```bash
ac create -f team-a-my-app-v1-mapping.yaml
```

Verify the metadata:

```bash
ac get imagestreams my-app -n team-a
ac get imagestreamtags my-app:v1 -n team-a -o wide
ac get imagestreamimages my-app@sha256:<digest> -n team-a -o yaml
```

Expected results:

- The `ImageStream` exists in the image namespace.
- The `ImageStreamTag` points to the expected digest and Registry v2 reference.
- The `ImageStreamImage` contains the expected manifest media type.

## Verify Migration

Switch the current context to Registry v2 mode:

```bash
ac config set-registry-mode modern
ac config get-registry-mode
```

Verify important business images and random samples:

```bash
ac get imagestreamtags my-app:v1 -n team-a -o wide
ac image info "$TARGET_REGISTRY_HOST/team-a/my-app@sha256:<digest>"
ac get images -n team-a
```

Pull a sample image from Registry v2:

```bash
nerdctl pull "$TARGET_REGISTRY_HOST/team-a/my-app:v1"
```

Acceptance criteria:

- Registry v2 can inspect or pull sample migrated images.
- `ImageStreamTag` references the expected digest and Registry v2 address.
- `ac get images` in `modern` mode shows migrated Image resources.
- Critical business namespaces can read their migrated images.
- Repeated metadata backfill does not produce unexplained failures.

## Switch Registry Mode

After verification, keep the current context in Registry v2 mode:

```bash
ac config set-registry-mode modern
ac config get-registry-mode
```

Use `--all-clusters` only after every cluster in the ACP session has completed Registry migration:

```bash
ac config set-registry-mode modern --all-clusters
```

Run dry-run image management commands to confirm that the backend has switched to Registry v2:

```bash
ac get images
ac delete images --repo team-a/my-app:v1
ac adm prune images
```

`ac delete images` and `ac adm prune images` are dry-run by default. Add `--confirm` only after reviewing the output.

## Roll Back

If workloads cannot read business images after cutover, switch the affected context back to legacy mode:

```bash
ac config set-registry-mode legacy
ac config get-registry-mode
```

Keep the following migration artifacts for troubleshooting and reruns:

- Image lists.
- Mirror mapping files.
- `ac image mirror` output.
- `ac image info` output.
- Generated `ImageStreamMapping` YAML files.
- `ac create -f` output.
- Migration verification output.

After the issue is fixed, continue from the image list. You do not need to rebuild the migration scope from the beginning.

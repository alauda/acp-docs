# Hardware accelerators

Use this section to choose the right accelerator path for container workloads, understand the ACP concepts that affect resource requests, and continue to the product documentation for installation, upgrade, compatibility, and troubleshooting.

Start from the scenario closest to your task. Vendor integrations prepare hardware, drivers, runtimes, and direct allocation. Sharing and scheduling products such as HAMi build on those integrations to provide sharing, virtualization, scheduling, or resource orchestration behavior.

## Choose a scenario

| If you want to... | Start here |
| --- | --- |
| Use NVIDIA GPUs with vendor-standard components | [GPU accelerator family](./families/gpu.mdx) and [NVIDIA vendor integration](./products/vendor-integrations/nvidia.mdx) |
| Run container workloads on Ascend NPUs with vendor-standard components | [NPU accelerator family](./families/npu.mdx) and [Ascend vendor integration](./products/vendor-integrations/ascend.mdx) |
| Share, virtualize, or schedule GPU or NPU devices across workloads | [HAMi sharing and scheduling path](./products/sharing-and-scheduling/hami.mdx) |
| Use Kubernetes DRA-based resource claims | [DRA mechanism](./mechanisms/dra.mdx) |
| Show accelerator resources in ACP quota pages or configure quota field metadata | [Accelerator Resource Quota](./mechanisms/resource-quota.mdx) |
| Use GPU passthrough for virtual machines | Use [Virtual Machine GPU passthrough](../virtualization/virtualization/virtual_machine/how_to/vm_gpust.mdx). This is a different path from container workload pGPU or vGPU usage. |

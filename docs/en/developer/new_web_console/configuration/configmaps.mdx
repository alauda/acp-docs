# ConfigMaps

In the **Applications** view, you can use **ConfigMaps** to store non-sensitive configuration data that workloads in the same namespace can reference.

## Prerequisites

Before working with ConfigMaps, ensure that:

- You can access the target project in the **Applications** view.
- The project is associated with the relevant cluster.
- The target namespace is available in the selected cluster.
- You understand which workloads will consume the configuration and how they consume it.

## Understand configuration scope

The ConfigMap page is opened from the project-scoped **Applications** view, but each ConfigMap is still a namespace-scoped resource in a specific cluster.

In practice, this means the list belongs to the current project scope, while create, update, detail, and workload-consumption checks still depend on the selected cluster and namespace.

A workload can only reference ConfigMaps from the same namespace.

## Open the ConfigMap list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **ConfigMaps**.

From the ConfigMap list, you can typically:

- View ConfigMaps in the current project scope
- Open ConfigMap details
- Create a ConfigMap
- Update an existing ConfigMap
- Delete a ConfigMap

## Create a ConfigMap

1. Open the ConfigMap list.
2. Click **Create**.
3. Select the target **Cluster** and **Namespace** if required by the page context.
4. Enter the ConfigMap name.
5. Configure the ConfigMap entries.
6. Click **Create**.

### Common entry patterns

Depending on the current UI behavior, the create or update page can support operations such as:

- Adding key-value entries one by one
- Bulk pasting multiple `key=value` lines
- Importing text files as entries
- Importing binary files where supported by the UI

When you use bulk paste, keep one `key=value` pair on each line. For example:

```text
APP_ENV=production
LOG_LEVEL=info
FEATURE_FLAG=true
```

Use ConfigMaps for non-sensitive runtime configuration. If the data contains credentials, tokens, or private keys, use **Secrets** instead.

## How workloads typically consume ConfigMaps

ConfigMaps are commonly consumed in workloads through patterns such as:

- Injecting all entries as environment variables
- Injecting a specific key as a single environment variable
- Mounting keys as files in a volume
- Reusing the same ConfigMap across multiple workloads in the same namespace

Before you submit the ConfigMap, confirm that the keys, expected file names, and value formats match the way the workload reads them.

## Update or delete a ConfigMap

Use the list page or details page actions to update or delete a ConfigMap.

:::warning
If a workload references a ConfigMap through environment variables or mounted files, updating or deleting that ConfigMap can affect workload behavior. Some changes require the workload Pods to be recreated before the updated configuration takes effect.
:::

Depending on the supported entry type, imported binary content can be more restrictive to edit after creation than regular text entries. Confirm the current UI behavior before using ConfigMaps for frequently changing binary content.

## Verify the result

After creating or updating a ConfigMap, verify that:

- The ConfigMap appears in the selected project and cluster context.
- The expected namespace is shown.
- The imported or entered keys appear as expected.
- The value format matches how the workload reads the configuration.
- Referencing workloads can use the configuration as intended.
- Any required Pod recreation or rollout has been completed if the referenced configuration is injected through environment variables.

After deleting a ConfigMap, verify that:

- The ConfigMap no longer appears in the list.
- No workload restart, recreate, or rollout now fails because the referenced ConfigMap is missing.

# Secrets

In the **Applications** view, you can use **Secrets** to store sensitive configuration data that workloads in the same namespace can reference.

## Prerequisites

Before working with Secrets, ensure that:

- You can access the target project in the **Applications** view.
- The project is associated with the relevant cluster.
- The target namespace is available in the selected cluster.
- You understand the sensitivity and intended use of the data you plan to store.

## Understand configuration scope

The Secret page is opened from the project-scoped **Applications** view, but each Secret is still a namespace-scoped resource in a specific cluster.

In practice, this means the list belongs to the current project scope, while create, update, detail, and workload-consumption checks still depend on the selected cluster and namespace.

A workload can only reference Secrets from the same namespace.

## Open the Secret list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **Secrets**.

From the Secret list, you can typically:

- View Secrets in the current project scope
- Open Secret details
- Create a Secret
- Update an existing Secret
- Delete a Secret

## Create a Secret

1. Open the Secret list.
2. Click **Create**.
3. Select the target **Cluster** and **Namespace** if required by the page context.
4. Select the Secret type.
5. Enter the required data.
6. Click **Create**.

### Common Secret types

Depending on the current UI behavior, the page can support Secret types such as:

- **Opaque** for general key-value sensitive data
- **Basic authentication** for username and password pairs
- **TLS** for certificates and private keys
- **SSH authentication** for SSH private keys or related authentication data
- **Image pull** credentials for private registries

Depending on the edit mode, the console can help convert the entered data into the Kubernetes Secret representation used in YAML.

## How workloads typically consume Secrets

Secrets are commonly consumed in workloads through patterns such as:

- Injecting a Secret key as an environment variable
- Injecting all supported keys as environment variables
- Mounting Secret data as files in a volume
- Using registry credentials through image-pull configuration

Before you submit the Secret, confirm that the target workload expects the selected Secret type and that the key names match the consuming configuration.

## Update or delete a Secret

Use the list page or details page actions to update or delete a Secret.

:::warning
If a workload references a Secret, updating or deleting that Secret can affect workload startup, image pulling, mounted files, or environment-variable injection. Some changes require Pods to be recreated before the updated configuration takes effect.
:::

Do not delete platform-generated or system-critical Secrets unless you explicitly understand the impact. This caution especially applies to authentication-related or service-account-related Secrets that can be required for platform behavior.

## Verify the result

After creating or updating a Secret, verify that:

- The Secret appears in the selected project and cluster context.
- The expected namespace is shown.
- The Secret type and entries match the intended use.
- Referencing workloads can consume the Secret as expected.
- Any required Pod recreation, rollout, or image-pull revalidation has been completed where relevant.

After deleting a Secret, verify that:

- The Secret no longer appears in the list.
- No workload restart, recreate, mount, or image pull now fails because the referenced Secret is missing.

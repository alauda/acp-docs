# Manage Application Resources

In the **Applications** view, you can review and manage resources associated with an application from the application details page.

## Understand application resource association

Application association determines which resources appear together in the application context for review and ongoing management.

Resources created as part of the application workflow are usually already associated with the application. Resources created separately can still work at runtime, but they might not appear in the application details until they are imported into the application association.

Importing resources into the application association can improve:

- Visibility of related resources from one application entry
- Version and rollback review when the environment records associated resources together
- Operational discoverability for networking, configuration, or storage dependencies
- Consistency when teams want the application details page to reflect the real business resource set

## Review associated resources

On the application details page, the resource area can group associated resources by kind and provide direct links to individual resources.

Use this area to:

- Review the resources currently associated with the application
- Filter associated resources by name
- Open linked resources for deeper inspection
- Understand which runtime, configuration, or supporting objects belong to the application context

If an expected resource is missing from this area, confirm whether it was created outside the application workflow and therefore needs to be imported rather than assuming the resource does not exist.

## Manage resources

When the resource management dialog is available, you can:

- Review the current associated resource list
- Filter resources by name
- Remove one or more associated resources
- Select a resource type
- Import additional resources into the application association
- View the YAML of an associated resource during review

Depending on the current UI behavior, supported importable resource types can include workload, network, storage, and configuration objects.

## What import or removal changes

Importing a resource usually changes how the resource is represented in the application management context rather than changing the underlying runtime definition by itself.

In practice, importing or removing resources can affect:

- Whether the resource appears in the application details view
- Whether the resource is reviewed together with the application boundary
- Whether history, export, or rollback-oriented workflows treat the resource as part of the application context when the environment supports that behavior

These actions do not automatically mean the runtime relationship between resources is created, removed, or rewritten.

## Remove associated resources

1. Open the resource management area.
2. Select one or more associated resources.
3. Click **Remove**.
4. Confirm the operation if required.

Removing a resource from the application association usually means disassociating it from the application rather than deleting the resource itself.

Before removal, confirm that the target resource is not still required for startup, configuration injection, networking, or storage continuity.

If multiple resources are operationally related, remember that removing one resource from the application association does not necessarily break the runtime relation between those resources. It mainly changes whether the resource appears as part of the application-managed context.

### Removal review points

Before removing resources, confirm that:

- The resource should no longer be treated as part of the intended application boundary.
- The resource is not being kept only for convenience while still serving a real runtime dependency.
- The resource is not a shared object whose disappearance from the application view would make later troubleshooting harder.
- The team understands that removal usually changes visibility and association rather than deleting the object.

## Import associated resources

1. Open the resource management area.
2. Select the target resource type.
3. Select one or more resources.
4. Click **Import Resources**.

Use import when you need an existing resource to appear within the application association context for visibility or ongoing management.

### Import review points

Before importing resources, confirm that:

- The resources belong to the same intended application context.
- The selected resources really support the business function of this application.
- You are not importing unrelated shared resources just because they are in the same namespace.
- The current resource state is suitable to be represented together with the application.

If the page limits how a resource can be associated, follow the current UI validation result rather than assuming all existing resources can be imported without restriction.

## When to import or remove resources

Typical reasons to import resources include:

- A Service, ConfigMap, Secret, or PVC was created separately but should appear in application-level review.
- A workload was introduced through YAML or another workflow and should now be visible from the application details page.
- Teams want history, export, or operational review to reflect a more complete application boundary.

Typical reasons to remove resources include:

- The resource is no longer part of the intended application boundary.
- The resource was associated by mistake.
- The resource remains valid in the namespace but should be managed outside the application context.

## Verify the result

After managing application resources, verify that:

- The associated resource list reflects the expected additions or removals.
- Resource grouping and filtering behave as expected.
- Imported resources appear under the application association.
- Removed resources no longer appear in the associated resource list.
- Related workloads still behave as expected if the affected resources are used at runtime.
- The resulting application boundary shown in the details page matches your management intent.
- The operation changed application-level visibility and association in the way you expected, without assuming that it deleted or rewired the runtime resource automatically.

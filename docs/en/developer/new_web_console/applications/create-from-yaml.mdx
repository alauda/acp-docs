# Create Applications from YAML

In the **Applications** view, you can create an application directly from YAML when you already have valid resource definitions for the target project, cluster, and namespace.

## When to use this workflow

Use **Create from YAML** when:

- You already maintain workload manifests.
- You need tighter control over object definitions than a form-based workflow provides.
- You want to submit one or more related resources together.
- You need to express fields that are easier to manage directly in YAML than through the form workflow.

If you mainly need to adjust common application parameters in the console, use [Create and Update Applications](./create-update) instead.

## Prerequisites

Before creating an application from YAML, ensure that:

- You can access the target project in the **Applications** view.
- The target cluster is associated with the current project.
- The target namespace is available in the selected cluster.
- The YAML content is valid and matches the target cluster and namespace context.
- Any referenced images, ConfigMaps, Secrets, PVCs, or related resources already exist, or are included correctly in the submitted YAML.

If you need to create or import a namespace first, see [Manage Namespaces](../namespaces/manage-namespaces).

## Create an application from YAML

1. In the **Applications** view, open the relevant application page.
2. Click **Create from YAML**.
3. Select the target **Cluster**.
4. Confirm or select the target **Namespace** in the metadata area.
5. Paste or edit the YAML content.
6. Review the visible metadata and the page validation result.
7. Click **Create**.

## Understand YAML behavior

When you create an application from YAML:

- The selected cluster determines the project context for the creation workflow.
- YAML content is validated before submission.
- If the YAML content cannot be parsed correctly, the console displays a format error and the submission does not continue.
- If the application name in the YAML content differs from the metadata value, the console can synchronize the visible name field with the YAML content.

Successful YAML validation only confirms that the content can be interpreted by the workflow. It does not guarantee that referenced images, storage, permissions, scheduling constraints, or cross-object dependencies will behave correctly at runtime after creation.

## Review points before submission

Before clicking **Create**, confirm that:

- The namespace in the YAML content matches the target namespace you intend to use.
- The image source is reachable from the target environment.
- Referenced ConfigMaps, Secrets, services, and storage objects exist or are included consistently.
- Labels, selectors, ports, and environment settings remain internally consistent across the submitted objects.
- The YAML does not unintentionally conflict with an existing object that has the same identity in the selected scope.
- Any scheduling constraints, such as node selection for architecture-specific images, still match the target cluster.

This is especially important when you submit multiple related resources together rather than a single workload object.

## What to include together

When you use this workflow, it is often safer to submit a coherent set of related resources together when they are tightly coupled.

Typical examples include:

- a workload and its Service
- a workload plus referenced ConfigMaps or Secrets created in the same submission
- a workload plus related ingress or routing resources when the environment expects them together

If you split tightly coupled resources across separate operations, verify that the first submission does not fail because the second resource is not yet available.

## Object consistency guidance

When you submit multiple objects in one YAML operation, review not only each object individually but also how the objects relate to one another.

Pay special attention to:

- workload selectors and Pod template labels
- Service selectors and target ports
- referenced Secret, ConfigMap, PVC, and Service names
- namespace metadata repeated across objects
- whether the same object identity already exists in the selected scope

A YAML file can be syntactically valid while still creating an unusable workload if the objects are logically inconsistent with each other.

## Verify the result

After creating an application from YAML, verify that:

- The application appears in the current project and cluster context.
- The application is associated with the expected namespace.
- The YAML content has been accepted without remaining format or submission errors.
- The created workloads and related resources are visible in the application details or resource lists.
- Referenced configuration, storage, and image settings resolve correctly at runtime.
- The resulting workloads become healthy rather than only being accepted by the submission workflow.

## Limitations and notes

- YAML-based creation still uses the selected project and cluster scope in the **Applications** view.
- Invalid or incomplete YAML content blocks submission.
- The visible metadata and the YAML content must remain logically consistent.
- YAML validation alone does not guarantee that all runtime dependencies are available in the target environment.
- If the YAML references private registries or cluster-specific capabilities, validate those dependencies before you submit.

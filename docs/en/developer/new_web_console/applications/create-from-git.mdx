# Create Applications from Git

In the **Applications** view, you can create an application from a Git repository by combining source retrieval, image build settings, workload settings, and optional network exposure in a single workflow.

## Prerequisites

Before creating an application from Git, ensure that:

- You can access the target project in the **Applications** view.
- The target project is associated with the relevant cluster.
- The target namespace is available in the selected cluster.
- Source-to-image or the required Git build capability is enabled and available in the selected cluster.
- You have permission to use the target build resources and any required credentials.

## When to use this workflow

Use **Create from Git** when the application should be built from repository source as part of the creation workflow rather than deployed from a prebuilt image or a pre-authored YAML manifest.

This workflow is usually a better fit when:

- The repository is the current source of truth for the application code.
- The target environment supports the required Git build capability.
- You want the initial build and workload creation to stay in one console-driven flow.

If you already have a tested image, use [Create and Update Applications](./create-update). If you already maintain resource manifests directly, use [Create Applications from YAML](./create-from-yaml).

## Open the Git-based creation page

1. In the **Applications** view, open the relevant application page.
2. Click **Create from Git**.
3. Select the target **Cluster**.

If the selected environment does not support the required Git build capability, the page can display an error banner and the creation workflow cannot continue normally.

## Configure the application

The Git-based creation page can include the following sections.

### Basic metadata

Configure the application metadata in the selected project and cluster context, including:

- Namespace
- Name
- Display Name

Confirm the namespace carefully before continuing. Repository access, referenced Secrets, and the resulting workload all need to align with the selected cluster and namespace context.

### Code repository

Provide the source repository information, including:

- Repository URL
- Optional revision identifier
- Optional source subdirectory
- Optional credential secret for repository access

The console validates the repository URL during the workflow and can display validation progress, success, warning, or error feedback.

Review this section carefully because it determines which source tree is actually built. A repository validation warning does not always block the workflow, but it can indicate that the detected repository structure, revision, or access method may not match the selected build behavior.

If the repository requires authentication, ensure that the referenced credential Secret is valid in the target namespace. An accessible repository URL alone does not guarantee that the build system can retrieve private source content during execution.

### Build settings

Configure the build settings, including:

- Build image
- Build image version
- Builder image source settings
- Build type options exposed by the current environment

The available build images depend on the current environment. If no matching build image is available, the page can display an empty state.

The selected build settings determine how source code is transformed into a runtime image. Review them not only as build inputs but also as compatibility choices for the repository language, build process, and target runtime behavior.

The repository validation result can also affect the visible build-image guidance. A validation warning does not always block the workflow, but it indicates that the detected repository characteristics and the selected build environment may require additional review.

### Deployment

Configure the initial workload settings, such as:

- Application name
- Replica count
- Resource requests and limits
- Environment variables
- Referenced configuration sources
- Pod-level runtime settings

When you configure the workload section, apply the same workload-planning principles used for image-based applications: resource settings should match the expected runtime load, referenced configuration must already exist in the namespace, and probe settings should reflect the startup and readiness behavior of the built application.

Keep in mind that successful source retrieval and image build do not guarantee a healthy runtime workload. Runtime configuration, probes, resource limits, and referenced ConfigMaps or Secrets still determine whether the created application starts correctly after the image is built.

### Network

Configure service exposure for the application. Depending on the selected options, the page can include:

- Target port
- Internal service settings
- External access switch
- Ingress settings when external access is enabled

Review this section together with the application runtime behavior. Exposing the application externally is not only a visibility change; it also assumes that the container actually listens on the expected port and that the created routing resources match the workload design.

### Labels and annotations

Add metadata such as labels and annotations if needed.

Use this section for workload selection, automation, or policy-related metadata only when you understand how those labels and annotations are consumed in the target environment.

## Review points before submission

Before clicking **Create**, confirm that:

- The repository URL, revision, and optional subdirectory identify the intended source tree.
- Any required credential Secret is valid in the selected namespace.
- The selected build image and build method match the repository technology and expected output.
- Referenced ConfigMaps, Secrets, and runtime dependencies already exist when the workload expects them.
- The selected port and exposure settings match how the application actually serves traffic.
- The requested resources and health checks are realistic for the first startup of the built application.

## Submit the creation

After you finish the configuration:

1. Review the application settings.
2. Optionally enable **Build Immediately**.
3. Click **Create**.

When **Build Immediately** is enabled, the workflow can continue directly into build execution after the application object is created. This speeds up initial delivery, but it also means repository, build, and runtime issues can surface immediately in sequence rather than being separated into later manual steps.

## Verify the result

After creating an application from Git, verify that:

- The application appears in the current project and cluster context.
- The application is associated with the expected namespace.
- The build starts as expected if **Build Immediately** is enabled.
- The build completes successfully when the source and build configuration are valid.
- The created workload, service, and ingress resources are visible when configured.
- The resulting Pods start successfully and the runtime behavior matches the selected workload and exposure settings.

## Limitations and notes

- Git-based creation depends on the selected cluster and its available build capability.
- Repository validation can return warnings even when the workflow remains available.
- External access configuration is only shown when you enable it.
- Some build methods can appear in the UI but remain unavailable in the current environment.
- If the source repository requires authentication, the referenced credential Secret must be valid in the target namespace and cluster context.
- If the built application exposes network access, ensure the target service and ingress settings remain aligned with the created workload after submission.

# Create and Update Applications

In the **Applications** view, you can create and update applications within the current project and cluster context. The application workflow supports both form-based creation and YAML-based creation.

## Prerequisites

Before creating or updating an application, ensure that:

- You can access the target project in the **Applications** view.
- The project is associated with at least one cluster.
- At least one accessible namespace is available in the selected cluster.
- The target namespace already exists if you want to deploy the application into an existing namespace.
- The required image, credentials, configuration objects, and storage dependencies are available for the chosen workload model.

If you need to create or import a namespace first, see [Manage Namespaces](../namespaces/manage-namespaces).

Before starting, confirm that the current project is already associated with the target cluster and that the target namespace is available in that cluster context.

## Create an application

1. In the **Applications** view, select the target project.
2. Open the target application resource page.
3. Click **Create**.
4. Select the target **Cluster**.
5. Select the target **Namespace**.
6. Enter the application metadata and workload configuration.
7. Preview the configuration if the page provides a preview step.
8. Click **Create**.

### Understand cluster and namespace selection

Although the **Applications** view is project-scoped, application creation and update are still executed in a concrete cluster and namespace context.

When you create an application in the **Applications** view:

- You must select a cluster in the current project scope.
- The namespace list is loaded from the selected project and cluster.
- If you change the cluster, the previously selected namespace is cleared and you must select a namespace again.

This behavior helps ensure that the namespace always matches the selected cluster.

### Understand metadata fields

When you create an application, the metadata form can include:

- Namespace
- Name
- Display Name

The application name must meet the platform naming rules used for Kubernetes resources.

### Common workload configuration areas

Depending on the workload model and the enabled capabilities in the current environment, the creation workflow can include sections such as:

- Workload model selection
- Replica settings
- Update strategy
- Pod configuration
- Container configuration
- Resource requests and limits
- Volumes and volume mounts
- Ports
- Environment variables and referenced configuration
- Health checks
- Log collection settings
- Service and ingress configuration

Many of these fields follow the same workload configuration principles summarized in this new web console module. Use the workload-specific pages in this section when you need more guidance for a particular workload type.

### Workload model review points

Before you choose the workload model, confirm whether the application should behave more like:

- a stateless replicated service
- a stateful replicated workload with persistent identity or storage
- a node-level workload that should run broadly on matching nodes

This decision affects the available rollout, scaling, storage, and lifecycle options during creation.

### Image and dependency review points

Before clicking **Create**, confirm that:

- the image source is reachable from the target environment
- private registry credentials are already available when required
- any referenced ConfigMaps, Secrets, PVCs, or ServiceAccounts exist in the same namespace
- service exposure or ingress dependencies are configured consistently with the selected workload

In mixed-architecture environments, also confirm that the scheduling-related configuration matches the image architecture requirements when the workload should run only on a specific node type.

### Before you submit the creation

Before clicking **Create**, confirm that:

- The selected cluster and namespace match the intended deployment target.
- The image, workload type, and resource settings match the expected runtime behavior.
- Any referenced ConfigMaps, Secrets, PVCs, or service exposure settings already exist or are configured correctly in the current namespace.
- Health checks, ports, and environment variables reflect the application runtime requirements.
- Service and ingress relationships are internally consistent when the application exposes network traffic.

## Update an application

1. In the **Applications** view, open the target application.
2. Click **Update**.
3. Review the existing configuration in the update or preview page.
4. Modify the supported fields.
5. Click **Update**.

### Fields that cannot be changed during update

When you update an application in the **Applications** view, the following fields remain fixed in the UI:

- Cluster
- Namespace
- Name

This means that the update workflow is designed to modify the application configuration in place, rather than move the application to another cluster or namespace.

### Update preview behavior

The update workflow can provide a preview page where you can:

- Review the current application structure
- Edit supported configuration items
- Switch between form-based and YAML-based editing where available
- Add or update application components before submitting the final change

### Force-update considerations

If the environment exposes both normal update and force-style update behavior, use the more disruptive action only when you explicitly need broad runtime refresh behavior.

This can matter when configuration changes must be propagated to workloads that would not otherwise pick up the new state immediately.

## Verify the result

After creating or updating an application, verify that:

- The application appears in the current project and cluster context.
- The application is associated with the expected namespace.
- The workload status becomes available in the application list or details page.
- The resulting workloads reflect the expected image, resources, configuration, and exposure settings.
- No YAML format errors or submission errors remain.

## Limitations and notes

- Cluster and namespace availability depends on the current project scope and your permissions.
- Changing the selected cluster during creation resets the namespace selection.
- Update does not support changing the cluster, namespace, or application name.
- YAML-based operations require valid resource definitions before submission.
- If the application depends on private images, Secrets, or external storage resources, those dependencies must already be available and accessible in the target namespace.
- Some operational effects, such as ConfigMap or Secret changes referenced through environment variables, can require Pod recreation or a follow-up update flow before the new configuration becomes effective.

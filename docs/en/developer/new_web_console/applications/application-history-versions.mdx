# Application History and Versions

In the **Applications** view, you can use the **History** tab on the application details page to review historical versions, create version snapshots, compare changes, export versions, and roll back to an earlier revision when supported.

## Open the history tab

1. Open the target application details page.
2. Click the **History** tab.

## Understand history records

The history area helps you understand how the application configuration changed over time and whether the current state should be preserved, exported, or rolled back.

This is especially useful when you need to:

- Review what changed after a recent update
- Preserve an important state before a high-impact modification
- Compare the current state with an earlier revision
- Recover to a previously known-good configuration
- Export a stable revision instead of the latest mutable state

## Review history records

The history list can display information such as:

- Version number
- Creation time
- Creator
- Comment or change cause
- Change summary or resource diffs

For non-template applications, the page can also provide action menus for version operations.

The history table can also help you understand whether the current entry is:

- a normal version progression
- a rollback-derived state
- a manually created snapshot
- a version with no effective resource diff compared with the previous revision

Depending on the current environment, snapshot retention can be limited. If the page retains only a fixed number of recent entries, preserve the version you need before relying on it for later rollback or export operations.

## Create a version snapshot

If version snapshot creation is available:

1. Open the **History** tab.
2. Click **Create Version Snapshot**.
3. Enter the required information.
4. Confirm the operation.

Create a snapshot when the application state is important enough to preserve even if the next change is made outside the main UI update workflow or if you want a stable rollback point before a risky modification.

### Snapshot review points

Before creating a snapshot, confirm that:

- The current application state is the one you actually want to preserve.
- Related resources already reflect the desired configuration.
- The current image, ConfigMaps, Secrets, and service relationships are in a usable state.
- You are not preserving a known broken runtime state by mistake.

## View a specific version

1. Open the **History** tab.
2. Click the target version entry.
3. Review the stored YAML or diff view.

When the selected version is not the current version, the page can display a diff view against the latest revision.

Use this view to confirm whether the meaningful change is in:

- image or tag selection
- replica or workload settings
- configuration references such as ConfigMaps or Secrets
- traffic or exposure-related resources
- associated resource membership

## Roll back to a historical version

If rollback is available:

1. Open the **History** tab.
2. Open the action menu for the target version.
3. Click the rollback action.
4. Confirm the rollback.

Use rollback only when you are confident that the selected historical version still matches the intended runtime dependencies, such as referenced ConfigMaps, Secrets, images, and service topology.

If the application uses imported associated resources, confirm whether the rollback will also need those resources to match the selected historical state.

### Rollback review points

Before rollback, confirm that:

- The selected version is really the target recovery point rather than only the last visible stable-looking entry.
- External dependencies still exist and remain compatible with that version.
- The rollback will not unintentionally reintroduce a configuration that is obsolete for the current environment.
- The expected impact on running traffic or stateful behavior is acceptable.

## Export a historical version

If export is available:

1. Open the **History** tab.
2. Open the action menu for the target version.
3. Click the export action.
4. Follow the export workflow shown by the console.

Export is useful when a specific historical version, rather than the latest version, is the one that should be distributed, archived, or reused in another environment.

Depending on the available workflow, export behavior can differ between current-state YAML-oriented output and template-oriented packaging. Review the target format carefully before using the result as a redeployment artifact.

## Verify the result

After reviewing or operating on history records, verify that:

- The correct version entry is selected.
- The YAML or diff content matches the expected revision.
- Newly created snapshots appear in the history list when expected.
- Rollback or export actions complete successfully when used.
- The current application state reflects the selected rollback result when applicable.
- The resulting runtime behavior remains consistent with the restored or exported version.

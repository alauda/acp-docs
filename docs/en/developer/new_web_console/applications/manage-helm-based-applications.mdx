# Manage Helm-Based Applications

In the **Applications** view, the migrated application routes also expose workflows for Helm-based applications backed by **HelmRequest** resources.

## Understand the HelmRequest relationship

A Helm-based application in this context is managed through a **HelmRequest** resource. When the platform synchronizes that HelmRequest into an application view, the application details page can expose template-application behaviors such as validation alerts, values display, and Helm-oriented update actions.

This means the application page is the operational view, while HelmRequest remains the declarative chart-driven source of deployment intent.

Use this page together with the application details view to understand how Helm-based applications expose values, history, and HelmRequest-related warning states in the new web console.

## Before you begin

Before creating or updating a Helm-based application, ensure that:

- the target project, cluster, and namespace are correct
- the selected chart and chart version match the intended workload
- any referenced configuration resources already exist in the target namespace when the chart expects them
- you understand whether the chart introduces namespace-scoped or broader resources that may need extra review

## When to use the Helm-based workflow

Use this workflow when the application should continue to be managed through chart values and Helm-oriented reconciliation behavior rather than only through direct workload editing.

This is usually the better fit when:

- the chart remains the intended source of deployment structure
- updates should be expressed as value changes rather than ad hoc resource edits
- the application depends on chart-defined templates, dependencies, or release-oriented history

If you mainly need a direct workload-oriented creation flow, use [Create and Update Applications](./create-update) or [Create Applications from YAML](./create-from-yaml) where appropriate.

## Create a Helm-based application

If the HelmRequest creation route is available:

1. Open the Helm-based application creation workflow.
2. Configure the Helm request form.
3. Review chart name, chart version, and target workspace information.
4. Review the values or parameters exposed by the chart.
5. Click **Deploy**.

### Creation review points

Before deployment, confirm that:

- the chart values match the target environment
- resource requests and limits are appropriate for the cluster
- dependent resources such as Secrets, ConfigMaps, or storage classes are already available when required
- the chart does not unintentionally rely on cross-namespace behavior that the current workflow should avoid

## Update a Helm-based application

If the HelmRequest update route is available:

1. Open the target Helm-based application.
2. Open the update workflow.
3. Modify the supported Helm request settings.
4. Click **Update**.

Depending on the current HelmRequest source, some chart resource fields can be read-only during update.

Treat read-only chart resource fields as a signal that the current application state is still governed by an existing chart source or release definition. If a value cannot be changed here, avoid assuming that a manual application-side edit would remain consistent with the Helm-managed source of truth.

### Value review guidance

When you review or update chart values, focus especially on:

- image-related values
- replica or scaling-related values
- service exposure settings
- storage-related values
- references to Secrets, ConfigMaps, or other dependent resources

A value change can indirectly affect multiple rendered resources at once, so review the likely workload impact rather than only the field you edited.

## Review template-application details

On the details page for a template-based application, the page can provide additional behaviors such as:

- Validation or permission warning banners related to HelmRequest status
- A **Values** tab for reviewing effective chart values
- History records presented in the template-application context

When the page reports HelmRequest-related warnings, treat them as deployment-health signals rather than cosmetic messages. They can indicate permission problems, reconciliation failures, chart dependency issues, or invalid chart-related state that should be addressed before further updates.

## Typical validation and sync states

Depending on the current environment, Helm-based application details can expose states related to:

- initialization or chart retrieval
- validation of permissions or dependencies
- deployment synchronization

If one of these states fails, review the reported reason before retrying the deployment. The failure can come from chart loading, dependency resolution, permission limits, or reconciliation problems rather than from the application workload alone.

A visible values change or accepted update request does not by itself confirm that the Helm-managed resources have finished reconciling successfully. Always check the resulting application state after the update is processed.

## Verify the result

After creating or updating a Helm-based application, verify that:

- The application appears in the expected project, cluster, and namespace context.
- The Helm-related warning or validation state is acceptable.
- The **Values** and **History** areas reflect the current deployment state.
- The updated chart configuration is shown as expected.
- The resulting workloads and related resources become healthy in the target namespace.
- The reconciliation result matches the intended chart change rather than only showing that the update request was accepted.

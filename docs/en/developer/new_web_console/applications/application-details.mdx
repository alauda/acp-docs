# View Application Details

In the **Applications** view, you can open an application details page to inspect its current state, related resources, runtime logs, events, monitoring data, and associated operational actions.

## Open the application details page

1. In the **Applications** view, select the target project.
2. Open the **Applications** list.
3. Click the target application name.

## Understand the details page role

The application details page is the main operational entry for reviewing how the application is currently represented in the selected project, cluster, and namespace context.

Use this page when you need to:

- Confirm which resources are currently grouped under the application
- Review runtime state before deciding whether to update, roll back, or troubleshoot the application
- Compare configuration-oriented information with runtime-oriented signals such as Pods, logs, events, and monitoring
- Open follow-up operations such as history review, resource management, or alert-related actions

## Detail page tabs

The application details page can include the following tabs.

The exact tab set can vary by application type, enabled features, and current permissions. For example, template-based applications can expose Helm-oriented tabs or warnings that do not appear on regular applications.

### Detail Info

Use **Detail Info** to review the application summary and basic configuration.

Typical information can include:

- Application name and type
- Current project, cluster, and namespace context
- Labels or identifying metadata
- Related workload summary
- Basic health or status indicators

Use this tab first when you need to confirm that you opened the correct application before interpreting the rest of the runtime data.

### Topology

Use **Topology** to inspect how the application resources are connected and organized.

This view is especially useful when you need to understand:

- Which workloads, Services, routes, or supporting resources are visually associated
- Whether the displayed relationships match the intended traffic or dependency model
- Whether an expected resource is missing from the application association

If the topology view does not match the expected application shape, review both the application resource association and the underlying resource definitions rather than assuming the runtime problem is only in one workload.

### YAML

When YAML display is available, use this tab to review the current resource definition.

This is useful when you need to:

- Confirm the effective metadata and spec currently visible to the page
- Compare the current definition with a historical version or exported result
- Check whether an observed runtime issue is explained by the current configuration

Treat the YAML shown here as the currently visible definition in the selected context. If you need change history or rollback-oriented comparison, continue with the **History** tab.

### Pod

Use the **Pod** tab to review the Pods associated with the application.

Pay particular attention to:

- Pod phase and restart behavior
- Which workload created the Pod
- Whether node placement matches scheduling expectations
- Whether a runtime issue is isolated to one Pod or affects the application more broadly

Use the Pod tab together with logs and events when you need to narrow an application issue down to a specific runtime instance.

### Values

For template-based applications backed by HelmRequest, the details page can include a **Values** tab for reviewing the effective chart values.

Review this tab when you need to confirm whether a Helm-based application is currently using the intended values rather than relying only on the chart name or version label.

### History

Use the **History** tab to review application history records, version snapshots, diffs, and rollback actions.

This tab is most useful when you need to answer questions such as:

- What changed between versions
- Whether the current state came from a normal update or from rollback
- Which historical revision should be reviewed or exported

### Logs

Use the **Logs** tab to inspect real-time container logs for Pods selected by the application.

Typical log operations can include:

- Switching between Pods or containers
- Viewing previous logs for restarted containers
- Adjusting the displayed line count
- Enabling or disabling line wrapping
- Searching within the visible log buffer
- Viewing raw logs or exporting logs when supported by the page

Application-level log review is useful for fast investigation, but the log stream can still reflect one specific Pod or container at a time. When diagnosing an issue, confirm which Pod and container are currently selected before drawing conclusions about the whole application.

If the visible viewer appears truncated or filtered by page limits, prefer raw or exported logs when those actions are available.

### Events

Use the **Events** tab to review recent runtime or scheduling events related to the application.

Event records can help identify:

- Scheduling failures
- Image pull problems
- Probe failures
- Restart-related behavior
- Resource creation or reconciliation transitions

When reviewing events, pay attention to:

- The affected resource
- The event reason
- Whether the level is informational or warning-oriented
- Whether the same event is repeating over time

Events are especially useful for understanding why the platform or Kubernetes made a runtime decision. Keep in mind that event retention is time-limited in many environments, so older failures may no longer appear on the page.

### Monitoring

Use the **Monitoring** tab to inspect monitoring dashboards, metrics, and related visualization data when monitoring is available.

Depending on the available dashboards and monitoring components, the page can expose metrics such as:

- CPU utilization or usage
- Memory utilization or usage
- Network traffic or packet rate
- Disk throughput or related storage activity

Use these metrics to determine whether the application issue is more likely caused by:

- Resource saturation
- Abnormal traffic behavior
- Persistent memory pressure
- Storage or I/O bottlenecks

Monitoring is most effective when combined with the Pod, Logs, and Events tabs rather than used in isolation.

### Alarm

Use the **Alarm** tab to review alert rules associated with the application and to open related alarm operations.

This tab is useful when you need to confirm whether the application already has alert coverage for the behavior you are currently investigating.

## Related header actions

Depending on the application type and your permissions, the details page header can provide actions such as:

- Updating the display name
- Opening update-related actions
- Accessing template-application operations
- Opening resource management functions

When header actions differ between two applications, check whether the difference comes from application type, permissions, or the presence of Helm-related management features.

## Troubleshoot from the details page

When the application appears unhealthy, use the details page in this general order:

1. Confirm the selected project, cluster, and namespace.
2. Review the high-level state in **Detail Info**.
3. Check **Pod** status and restart behavior.
4. Use **Events** to identify orchestration or scheduling problems.
5. Use **Logs** to inspect application-level runtime output.
6. Use **Monitoring** to confirm whether the issue correlates with resource or traffic pressure.
7. Use **History** if you need to determine whether a recent change introduced the issue.

## Verify the result

When reviewing the application details page, verify that:

- The expected application is opened.
- The tabs relevant to the current application type are displayed.
- Pods, resources, logs, events, and monitoring data are visible when available.
- Header actions match your permissions and the application type.
- The observed runtime information is consistent across summary, Pod, event, log, and monitoring views.

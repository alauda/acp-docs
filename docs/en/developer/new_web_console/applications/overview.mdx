# Applications Overview

The **Overview** page in the **Applications** view provides a project-scoped dashboard for monitoring application health, resource usage, and alert activity across the clusters associated with the current project.

## Prerequisites

Before using the overview dashboard, ensure that:

- You can access a project in the **Applications** view.
- The project is associated with at least one cluster.
- Monitoring components are installed and healthy in the associated clusters if you want to view resource usage, trend charts, and alert data.

Before using the overview dashboard, confirm that the current project is associated with at least one cluster and that the monitoring stack is available where dashboard data is expected.

## Understand dashboard scope

The overview page is project-scoped, but the displayed runtime and monitoring data can aggregate across the clusters associated with the current project.

This means the page helps you evaluate application health at the project level first, then drill into namespace, workload, or cluster-specific issues from the affected entries.

## Open the overview dashboard

1. In the web console, switch to the **Applications** view.
2. Select the target project.
3. In the left navigation pane, click **Overview**.

## Dashboard sections

The overview dashboard can display the following sections.

### View project details

The **Detail** card provides basic information about the current project, including:

- Project status
- Display name
- Project administrator
- Log policy

Use this section to quickly confirm the project state and the effective log retention strategy.

### View namespace and member counts

The summary card shows:

- The number of namespaces in the current project
- The number of project members

If the namespace count is greater than 0, you can click the number to open the namespace list.

### View application status

The **Applications** card summarizes the status of applications in the current project. Depending on the runtime state of the applications, the chart can include statuses such as:

- Running
- Partial Running
- Pending
- Failed
- Stopped
- No Workloads

Use this chart to identify whether application issues are isolated or widespread in the current project.

### View workload distribution

The **Workloads** section summarizes workload health by resource type, such as:

- Deployments
- StatefulSets
- DaemonSets

This section helps you compare workload status across the main workload types in the current project.

### View pod trends

The **Pods** section displays pod trends for the current project over time. The chart can include the following categories:

- Running
- Abnormal
- Others
- Total

Use this chart to identify workload fluctuations or an increasing number of abnormal pods.

### View resource usage

The **Resources** section summarizes project-level resource usage and quota allocation.

The card supports the following resource types:

- CPU
- Memory

For each resource type, the dashboard can display:

- **Usage Rate**, which reflects current usage against the project quota
- **Quota Rate**, which reflects the allocated namespace quota against the project quota

You can switch between **CPU** and **Memory** to view the corresponding resource details and trend chart.

### View utilization rankings

The **Utilization Top 5** section highlights the highest-utilization entries in the current project. You can switch the ranking between **CPU** and **Memory**.

Use this section to identify which workloads or project areas are consuming resources most aggressively and may require further investigation.

### View alert trends

The **Alerts** section displays alert trends for the current project. This chart focuses on higher-severity alerts, such as:

- Critical
- High

Use this section to determine whether current issues are isolated events or part of a broader trend.

## Understand warning banners and data availability

The overview dashboard can display warning or error banners when project or monitoring conditions affect the displayed data.

### Project is not associated with any cluster

If the current project is not associated with any cluster, the page displays a warning banner indicating that the project is not ready.

In this case, monitoring-dependent sections cannot provide meaningful project data.

### Business clusters are abnormal

If one or more associated business clusters are abnormal, the page can display an error banner indicating that some data may be inaccurate.

### Monitoring data is incomplete or unavailable

If monitoring components are not installed or are unhealthy, the page can display warnings indicating that only partial data is available.

In this case:

- Trend charts can be incomplete.
- Resource statistics can exclude some clusters.
- Alert data can be incomplete.

### Unlimited project quota

If the effective project quota for a resource is unlimited, some trend graphs cannot be displayed meaningfully for that resource.

## Time filters and refresh behavior

Some metric-based sections provide their own time filters. For example, resource trends, pod trends, and alert trends can use independent time ranges.

Because these filters are chart-specific, changing the time range in one chart does not necessarily change the time range in another chart.

## Verify the displayed data

After opening the overview dashboard, verify that:

- The page opens in the **Applications** view for the selected project.
- The project detail card shows the correct project information.
- Namespace and member counts match the current project scope.
- Monitoring-dependent charts display data when monitoring is available.
- Warning banners are shown only when cluster or monitoring conditions affect the results.

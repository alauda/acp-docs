# Manage Namespaces

The **Applications** view provides namespace management capabilities within the current project scope. You can view namespaces, create a namespace, import an existing namespace, update namespace settings, and review namespace details.

## When to use this page

Use this page when you need to:

- Prepare a namespace before creating or moving application workloads into it
- Bring an existing namespace under the governance of the current project
- Review or adjust quota, policy, metadata, or namespace-level operational settings
- Decide whether a namespace should stay associated with the current project or be removed from it

## Prerequisites

Before managing namespaces in the **Applications** view, ensure that:

- You can access the target project.
- The target project is associated with the relevant cluster.
- You have the required permissions to view or modify namespaces.

## Understand project and cluster scope

The **Applications** view is project-scoped, but each namespace still belongs to a concrete cluster.

In practice, this means:

- The namespace list is shown in the current project context.
- A namespace can only be created or imported into a cluster that is already associated with the current project.
- Namespace quotas, policies, and workload behavior still take effect in the real cluster where the namespace exists.

## Open the namespace list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **Namespaces**.

From the namespace list, you can typically:

- View namespace entries
- Create a namespace
- Import a namespace
- Open namespace details
- Run supported actions on a namespace

## Create a namespace

1. Open the namespace list.
2. Click **Create Namespace**.
3. Select the target **Cluster**.
4. Configure the namespace information.
5. Configure additional namespace settings as needed.
6. Click **Create**.

### Namespace naming behavior

When you create a namespace in the **Applications** view, the namespace name can follow project-oriented naming rules in the current environment.

Before submission, confirm that the final namespace name matches your project governance expectations, because namespace names are not typically changed later.

### Create page sections

The namespace create page can include sections such as:

- **Basic Information**
- **Resource Quota**
- **Container Limit Range**
- **Pod Security Policy** or equivalent admission-policy settings
- **More Configurations**

The **More Configurations** section can be used to add labels and annotations.

These sections are important because namespace creation in ACP is not only a naming step. It also defines the initial governance boundary for resource usage, container defaults, security behavior, and metadata that downstream workloads can inherit or reference.

### Resource quota guidance

When you configure a namespace quota, review:

- CPU and memory requests
- CPU and memory limits
- Pod count
- Storage quotas where the environment exposes them
- Any extended or custom quota types exposed by the current cluster capabilities

In project-scoped workflows, namespace quotas are normally bounded by what the project can already use in the target cluster. If an available quota is exhausted, namespace creation or later workload scheduling can fail.

### Container limit range guidance

When you configure a container limit range, review:

- default requests
- default limits
- maximum allowed values

These values mainly affect workloads created after the configuration is applied or updated. If the environment uses overcommit or request defaults, review how those defaults affect later workload forms.

### Security policy guidance

When the page exposes namespace-level pod security settings, review whether the namespace should use stricter or more permissive admission behavior.

In practice, stronger security settings can block Pods that rely on elevated privileges, host access, or incompatible security contexts. Confirm the expected workload requirements before tightening the policy.

## Import a namespace

1. Open the namespace list.
2. Click the dropdown button next to **Create Namespace**.
3. Click **Import Namespace**.
4. Select the target **Cluster**.
5. Select an existing namespace.
6. Review the imported namespace settings.
7. Click **Import**.

Use this workflow when you want to bring an existing namespace under the governance of the current project.

### Import review points

Before importing a namespace, confirm that:

- the namespace is in a cluster already associated with the current project
- the namespace is not still governed by another project when exclusive association rules apply
- the namespace's existing workloads, quotas, and policies are acceptable under the target project

Importing a namespace changes governance and visibility expectations, not just list membership.

## Update a namespace

1. Open the namespace list or the target namespace details page.
2. Click **Update** or the corresponding namespace action.
3. Modify the supported settings.
4. Click **Update**.

### Fields that remain fixed during update

When you update a namespace in the **Applications** view:

- The cluster cannot be changed.
- The namespace name cannot be changed.

### What you can update

Depending on the available actions and your permissions, you can update settings such as:

- Resource quota
- Container limit range
- Pod security policy
- Labels and annotations
- Display name
- Overcommit-related settings when the environment exposes them

When you update these settings, consider the impact on existing workloads in the namespace. Changes to quota or limit policies can affect future scheduling, while security-related changes can affect whether Pods are admitted or how they must be configured.

Changes to namespace-level request or overcommit behavior usually affect newly created or recreated Pods more directly than already running Pods.

## View namespace details

Open a namespace to access its detail page. Depending on the current environment, the detail page can include tabs such as:

- **Detail Info**
- **Namespace Member**
- **Container Limit Range**
- namespace-related policy or quota areas

From the detail page, you can review basic namespace information and perform supported management actions.

### Namespace member considerations

If the page exposes namespace member management, the member list can be used to review who has namespace-scoped access.

Depending on the current environment, member operations can include importing project members, adding eligible external identities, or removing namespace access from specific users.

## Remove a namespace or delete a namespace

The **Applications** view distinguishes between removing a namespace from a project and deleting the namespace itself.

### Remove Namespace

Use **Remove Namespace** to disassociate the namespace from the current project.

This action removes the project association, but does not necessarily delete the namespace resource from the cluster.

### Delete Namespace

Use **Delete Namespace** to delete the namespace resource itself.

:::warning
Deleting a namespace removes the namespace resource from the cluster and can delete or terminate the workloads and namespaced resources inside it. Use this action only when you intend to remove the namespace itself, not just its association with the current project.
:::

## Verify the result

After creating, importing, or updating a namespace, verify that:

- The namespace appears in the namespace list for the current project.
- The namespace is associated with the expected cluster.
- The updated quota, limit range, security policy, or metadata is displayed correctly.
- The namespace detail tabs are available as expected.
- Workloads created or managed afterward can use the namespace as intended within the configured quota and policy boundaries.

After removing or deleting a namespace, also verify that:

- The resulting project association or cluster state matches your actual intent.
- No dependent application workflow is now blocked because the namespace is missing or no longer associated with the project.

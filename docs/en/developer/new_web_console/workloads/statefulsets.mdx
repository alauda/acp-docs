# StatefulSets

In the **Applications** view, you can use **StatefulSets** to manage stateful workloads that require stable identity, ordered updates, or persistent storage in the current project and cluster context.

## Prerequisites

Before working with StatefulSets, ensure that:

- You can access the target project in the **Applications** view.
- The project is associated with the relevant cluster.
- The target namespace is available in the selected cluster.
- The required storage classes, persistent volume settings, and container images are available when needed.
- You understand whether the workload also depends on a headless Service, stable DNS identity, or per-replica storage.

## Understand workload scope

The StatefulSet page is reached from the project-scoped **Applications** view, but each StatefulSet still belongs to a specific cluster and namespace.

When you review identity, storage, and rollout behavior, confirm the selected cluster and namespace before assuming the state applies across the whole project.

## When to use a StatefulSet

Use a StatefulSet when the workload depends on per-replica identity, stable naming, ordered lifecycle behavior, or persistent storage that should stay associated with a specific replica.

A StatefulSet is usually the better fit when:

- each replica should keep a stable ordinal identity
- the workload depends on stable network naming or a related headless Service
- each replica needs its own persistent volume claim
- startup, shutdown, or update order matters to the application

If the workload can be treated as interchangeable stateless replicas, review [Deployments](./deployments) instead.

## Open the StatefulSet list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **StatefulSets**.

From the StatefulSet list, you can typically:

- View StatefulSets in the current project scope
- Open StatefulSet details
- Create a StatefulSet
- Create a StatefulSet from YAML
- Update an existing StatefulSet
- Delete a StatefulSet

## Create a StatefulSet

1. Open the StatefulSet list.
2. Click **Create**.
3. Select the target **Cluster** if required by the page context.
4. Select the target **Namespace**.
5. Configure the StatefulSet parameters.
6. Click **Create**.

### Common configuration areas

The create or update page can include configuration areas such as:

- Basic information
- Replica settings
- Update strategy
- Volume claim templates
- Pod configuration
- Container configuration
- Volumes, ports, and environment settings

In StatefulSet workflows, storage and identity settings usually matter as much as container settings. Review them together rather than treating storage as an optional add-on to the workload definition.

### Stateful workload review points

Before creating or updating a StatefulSet, review the following carefully:

- Whether each replica requires stable identity
- Whether persistent storage is required for each replica
- Whether the workload depends on ordered startup, shutdown, or scaling behavior
- Whether the referenced storage settings are available in the target environment
- Whether configuration changes could affect data continuity or replica replacement behavior
- Whether the workload also expects a stable network identity through a related Service

### Volume claim template guidance

When the page exposes per-replica storage configuration, confirm:

- which StorageClass is used
- the requested capacity for each replica
- the access mode expected by the application
- whether each Pod should receive its own PVC

A StatefulSet commonly creates one PVC per replica from the configured template. Before scaling the workload, confirm that the storage backend can satisfy the added claims.

Also confirm whether storage should outlive the Pod or workload lifecycle. A PVC that remains after replica changes can preserve data usefully, but it can also preserve stale state if the later workload design is different.

### Update strategy guidance

When rolling updates are supported, review whether the workload should:

- update all replicas progressively
- update only higher ordinals first when partition-style behavior is available

For stateful workloads, image and configuration changes can require a more conservative rollout strategy than a stateless Deployment.

## Create a StatefulSet from YAML

1. Open the StatefulSet list.
2. Click **Create from YAML**.
3. Select the target **Cluster** and **Namespace**.
4. Paste or edit the YAML content.
5. Click **Create**.

## Update a StatefulSet

1. Open the StatefulSet list or a StatefulSet details page.
2. Click **Update**.
3. Modify the supported configuration.
4. Click **Update**.

Before submitting an update, verify that the change will not break ordinal identity, attached storage expectations, or application quorum behavior.

A change that looks small at the form level can still be high impact if it alters storage usage, peer discovery behavior, or startup ordering.

## View StatefulSet details

Open a StatefulSet to review its runtime state and related operational data. Depending on the available integrations, the details area can include sections such as:

- Details
- Topology
- Logs
- Events
- Monitoring
- Custom charts or alerts

## Operational considerations

StatefulSet deletion and scaling can have storage consequences that differ from stateless workloads.

Before deleting a StatefulSet, confirm:

- whether the Pods can be removed safely
- whether the PVCs should be retained or cleaned up separately
- whether downstream applications depend on the persistent data that remains

Before scaling down, confirm that the removed replicas do not still hold required data or cluster membership.

Deleting the workload object does not automatically mean that the corresponding persistent state is no longer needed. Always treat storage cleanup as a separate decision from workload removal.

## Verify the result

After creating or updating a StatefulSet, verify that:

- The StatefulSet appears in the selected project and cluster context.
- The expected namespace is shown.
- The desired replicas and storage configuration are applied.
- Related Pods are created in the expected order when applicable.
- Related PVCs exist as expected when per-replica storage is configured.
- No rollout or storage-related errors remain.
- The resulting workload still matches the intended identity and persistence behavior.

After deleting a StatefulSet, verify that:

- The StatefulSet no longer appears in the list.
- Any remaining PVC or storage cleanup state matches your intended outcome.
- Any retained storage can still be accounted for correctly before later reuse or deletion.

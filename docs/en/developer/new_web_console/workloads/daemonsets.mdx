# DaemonSets

In the **Applications** view, you can use **DaemonSets** to manage workloads that are intended to run on matching nodes across the selected project and cluster context.

## Prerequisites

Before working with DaemonSets, ensure that:

- You can access the target project in the **Applications** view.
- The project is associated with the relevant cluster.
- The target namespace is available in the selected cluster.
- The required container image and node scheduling conditions are available.
- You understand whether the workload should run on all eligible nodes or only on a constrained subset.

## Understand workload scope

The DaemonSet page is reached from the project-scoped **Applications** view, but each DaemonSet operation is still tied to a specific cluster and namespace.

Because DaemonSets can expand broadly across matching nodes, always confirm the selected cluster before assuming the visible workload state applies everywhere in the current project.

## When to use a DaemonSet

Use a DaemonSet when the workload should run broadly across eligible nodes rather than as a small replicated application.

This model is commonly more appropriate when:

- the workload provides node-level functions such as collection, networking, or local agents
- each matching node should normally run one Pod from the workload
- node eligibility is controlled through selectors, affinity, or tolerations rather than replica count

If only a limited number of interchangeable application replicas are needed, review [Deployments](./deployments) instead.

## Open the DaemonSet list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **DaemonSets**.

From the DaemonSet list, you can typically:

- View DaemonSets in the current project scope
- Open DaemonSet details
- Create a DaemonSet
- Create a DaemonSet from YAML
- Update an existing DaemonSet
- Delete a DaemonSet

## Create a DaemonSet

1. Open the DaemonSet list.
2. Click **Create**.
3. Select the target **Cluster** if required by the page context.
4. Select the target **Namespace**.
5. Configure the DaemonSet parameters.
6. Click **Create**.

The create or update page can include workload, pod, container, storage, network, and environment configuration areas similar to other workload pages.

### Node-wide rollout review points

Before creating or updating a DaemonSet, confirm that:

- The workload is intended to run broadly across eligible nodes
- Node selection behavior matches your expectation
- Resource requests are appropriate for repeated per-node placement
- The update behavior will not create unnecessary disruption across the cluster
- The workload really needs node-wide deployment rather than a smaller scoped model

### Scheduling guidance

When the page exposes scheduling controls, review:

- node selectors or affinity rules
- whether taints or tolerations affect the intended node set
- whether some nodes are intentionally excluded because of readiness or schedulability constraints

A DaemonSet usually results in one Pod per eligible node. If the observed Pod count is lower than expected, recheck node qualification rather than only the workload definition.

Scheduling settings are effectively coverage rules for the workload. A small selector or toleration mistake can silently expand or shrink the node set much more broadly than a normal replicated workload change.

### Update strategy guidance

When rollout controls are available, focus especially on how many DaemonSet Pods can be unavailable during update.

For node-wide operational agents, a too-aggressive rollout can reduce visibility or functionality across many nodes at once.

## Create a DaemonSet from YAML

1. Open the DaemonSet list.
2. Click **Create from YAML**.
3. Select the target **Cluster** and **Namespace**.
4. Paste or edit the YAML content.
5. Click **Create**.

## Update a DaemonSet

1. Open the DaemonSet list or a DaemonSet details page.
2. Click **Update**.
3. Modify the supported configuration.
4. Click **Update**.

Changes to images, node selection rules, tolerations, or Pod configuration can affect cluster-wide placement and update behavior. Review the likely node coverage impact before submitting the change.

## View DaemonSet details

Open a DaemonSet to review workload state and related operational data. Depending on the available integrations, the details area can include sections such as:

- Details
- Topology
- Logs
- Events
- Monitoring
- Custom charts or alerts

## Troubleshoot common DaemonSet issues

If the DaemonSet does not reach the expected node coverage, check the following first:

- nodes excluded by selectors, affinity, or toleration mismatch
- nodes that are not ready or not schedulable
- image pull failures or missing credentials
- per-node resource pressure preventing Pod startup
- rollout settings leaving too many Pods unavailable at once

## Verify the result

After creating or updating a DaemonSet, verify that:

- The DaemonSet appears in the selected project and cluster context.
- The expected namespace is shown.
- Pods are created on the intended nodes.
- No rollout or scheduling errors remain.
- The resulting node distribution matches the intended workload scope.
- Cluster-wide visibility or agent coverage remains acceptable after update when the workload provides node-level operational functions.

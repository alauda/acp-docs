# Pods

In the **Applications** view, you can use **Pods** to inspect running workload instances, review runtime details, view logs, and perform selected operational actions in the current project and cluster context.

## Understand workload scope

The Pod page is opened from the project-scoped **Applications** view, but each Pod still belongs to a concrete cluster and namespace and is usually tied back to a higher-level workload.

When you inspect logs, events, or delete a Pod, confirm the cluster, namespace, and source workload before treating the observed state as a project-wide issue.

## Open the Pod list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **Pods**.

From the Pod list, you can typically:

- View Pods in the current project scope
- Open Pod details
- Review Pod status, restart count, node placement, and source workload
- Delete Pods when operationally appropriate

## View Pod details

Open a Pod to review runtime information. Depending on the available integrations, the details area can include sections such as:

- Details
- YAML
- Configuration
- Logs
- Events
- Monitoring

### Common runtime information

The Pod list or details page can display information such as:

- Resource requests and limits
- Source workload
- Restart count
- Node
- Service account

These fields are especially useful when you need to determine whether an issue comes from workload ownership, node placement, repeated restarts, or incorrect resource assumptions.

### Review points

When checking a Pod, pay attention to:

- whether the Pod is running, pending, completed, or repeatedly restarting
- whether the source workload explains why the Pod was created and whether it will be recreated automatically
- whether node placement matches scheduling expectations
- whether logs, events, and configuration details point to image, probe, permission, or resource problems

## Delete a Pod

Delete a Pod only when you understand the operational impact.

In many cases, Pod deletion is an operational recovery action rather than a configuration change. If the Pod is owned by a higher-level controller, the platform can recreate it automatically after deletion.

Typical use cases include:

- Recovering a Pod that is stuck in an abnormal state after the underlying issue has been resolved
- Cleaning up completed or evicted Pods when they are no longer needed for troubleshooting
- Forcing recreation of a managed Pod to pick up corrected runtime state when the owning controller will recreate it

Delete the Pod only after deciding whether the real fix belongs in the Pod's owner workload. If the issue comes from image, configuration, probe, scheduling, or permission errors, deleting the Pod alone usually does not correct the underlying cause.

### What Pod deletion changes

Pod deletion normally affects the current runtime instance, not the desired state stored in a higher-level controller.

In practice:

- a controller-managed Pod can be recreated automatically
- a standalone Pod might not be recreated at all
- logs, events, or current failure state tied to the deleted Pod can become harder to inspect afterward

:::warning
Deleting a Pod can affect running services or diagnostic visibility. Use this action carefully.
:::

## Troubleshoot common Pod issues

If the Pod is unhealthy, check the following first:

- repeated restarts that can indicate probe or application failures
- pending state caused by missing resources, selectors, or storage
- image pull errors caused by invalid image references or missing credentials
- permission or admission failures caused by security settings or missing service account access
- configuration issues caused by missing ConfigMaps, Secrets, or mounted paths

Use the Pod details, logs, events, and source workload together to decide whether the real fix belongs in the Pod's owner workload rather than on the Pod object itself.

## Verify the result

When reviewing or deleting a Pod, verify that:

- The Pod appears in the selected project and cluster context.
- The displayed runtime information matches the current workload state.
- Logs, events, and monitoring data are available when expected.
- If you delete a Pod, the resulting workload behavior matches your expectation.
- If the Pod is controller-managed, the replacement Pod is recreated and behaves as intended.
- If the Pod is standalone, its removal does not unexpectedly eliminate the only running instance you still needed.

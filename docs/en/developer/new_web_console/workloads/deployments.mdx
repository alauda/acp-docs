# Deployments

In the **Applications** view, you can use **Deployments** to manage stateless application workloads in the current project and cluster context.

## Prerequisites

Before working with Deployments, ensure that:

- You can access the target project in the **Applications** view.
- The project is associated with the relevant cluster.
- The target namespace is available in the selected cluster.
- You can access the required container image and related credentials if needed.
- Any referenced ConfigMaps, Secrets, PVCs, or ServiceAccounts already exist if the workload depends on them.

## Understand workload scope

The workload pages in this module are opened from a project-scoped application-management view, but each workload still belongs to a specific cluster and namespace.

Use the list page to find workloads in the current project scope, then confirm the concrete cluster and namespace before you create, update, or troubleshoot a specific workload.

## When to use a Deployment

Use a Deployment when the workload is primarily stateless and should be updated through replicated Pod replacement rather than stable identity or per-replica persistent state.

A Deployment is usually the better fit when:

- replica identity does not need to remain stable across replacement
- the workload should scale horizontally through interchangeable Pods
- rolling update behavior is preferred for application delivery

If the workload depends on stable ordinal identity or per-replica persistent storage, review [StatefulSets](./statefulsets) instead.

## Open the Deployment list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **Deployments**.

From the Deployment list, you can typically:

- View Deployments in the current project scope
- Open Deployment details
- Create a Deployment
- Create a Deployment from YAML
- Update an existing Deployment
- Delete a Deployment

## Create a Deployment

1. Open the Deployment list.
2. Click **Create**.
3. Select the target **Cluster** if required by the page context.
4. Select the target **Namespace**.
5. Configure the Deployment parameters.
6. Click **Create**.

### Common configuration areas

The create or update page can include configuration areas such as:

- Basic information
- Replica settings
- Update strategy
- Pod configuration
- Container configuration
- Volumes, ports, and environment settings
- Health checks and log settings

These areas do not all carry the same operational weight. Replica count and update strategy mainly influence availability during rollout, while container, probe, configuration, and volume settings determine whether the new Pod revision can actually start and serve traffic correctly.

### Deployment review points

Before submitting a Deployment, review the following carefully:

- Whether the replica count matches the intended availability target
- Whether the selected image and pull credentials are valid in the target cluster
- Whether referenced ConfigMaps, Secrets, or PVCs exist in the same namespace
- Whether resource requests and limits are appropriate for the workload
- Whether readiness and liveness probes match the application's real startup and health behavior
- Whether the rollout strategy is safe for the workload and traffic pattern

### Update strategy guidance

When the page exposes rollout controls, focus on settings such as:

- **Max surge** for how many extra Pods can be created during rollout
- **Max unavailable** for how many Pods can be unavailable during rollout

In practice:

- Prefer conservative values when the workload is sensitive to capacity loss.
- Avoid combinations that would make rollout progress impossible.
- Recheck rollout health after changing images, probes, resources, or environment configuration.

These settings control how much disruption the rollout can introduce. A rollout that is too aggressive can reduce available capacity, while a rollout that is too restrictive can slow recovery or prevent progress when the new Pods do not become ready quickly.

## Create a Deployment from YAML

1. Open the Deployment list.
2. Click **Create from YAML**.
3. Select the target **Cluster** and **Namespace**.
4. Paste or edit the YAML content.
5. Click **Create**.

Use this workflow when you already have valid Deployment manifests and want to create the workload directly from YAML.

Prefer YAML especially when the workload requires advanced manifest control or when the Deployment should be created together with tightly coupled resources such as a Service or supporting configuration objects.

## Update a Deployment

1. Open the Deployment list or a Deployment details page.
2. Click **Update**.
3. Modify the supported configuration.
4. Click **Update**.

Changes to images, probes, environment variables, ports, volume mounts, or resource settings can trigger a new rollout. Confirm that the updated Pod template is compatible with the existing runtime behavior before submitting.

A successful update request does not by itself mean the rollout is healthy. The real effect appears only after the new Pod revision schedules correctly, becomes ready, and replaces the previous revision without leaving the workload unavailable.

## View Deployment details

Open a Deployment to review information such as workload status and related operational data. Depending on the available integrations, the details area can include sections such as:

- Details
- Topology
- Logs
- Events
- Monitoring
- Custom charts or alerts

## Troubleshoot common Deployment issues

If the Deployment does not become healthy after creation or update, check the following first:

- Image pull failures caused by invalid image addresses or missing pull credentials
- Probe failures caused by incorrect endpoints, ports, or startup timing
- Unschedulable Pods caused by missing resources, selectors, or affinity constraints
- Configuration errors caused by missing ConfigMaps, Secrets, or mounted paths
- Rollout stalls caused by unavailable replicas or broken new Pod revisions

When available in the UI, use **Events**, **Logs**, and related Pod status views before changing the Deployment again.

## Verify the result

After creating or updating a Deployment, verify that:

- The Deployment appears in the selected project and cluster context.
- The expected namespace is shown.
- The desired replica state is reached.
- Related Pods become available.
- No update or rollout errors remain.
- The new Pod template actually reflects the expected image, configuration, and resources.
- Logs, events, and monitoring views reflect a healthy rollout when those integrations are available.

After deleting a Deployment, verify that:

- The Deployment no longer appears in the list.
- The managed Pods and ReplicaSet behavior match the expected deletion result.
- Any separate Service, ConfigMap, Secret, or PVC that should remain is still present if it was not meant to be removed with the workload.

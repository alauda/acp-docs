# CronJobs

In the **Applications** view, you can use **CronJobs** to manage scheduled workloads that run repeatedly according to configured execution policies in the current project and cluster context.

## Prerequisites

Before working with CronJobs, ensure that:

- You can access the target project in the **Applications** view.
- The project is associated with the relevant cluster.
- The target namespace is available in the selected cluster.
- The required container image is available.
- You understand whether the task should run on a schedule, by manual trigger, or both.

## Understand workload scope

The CronJob page is opened from the project-scoped **Applications** view, but each CronJob is still created, updated, and executed in a concrete cluster and namespace context.

When you review schedules or immediate executions, confirm the selected cluster and namespace before interpreting downstream Job results.

## Open the CronJob list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **CronJobs**.

From the CronJob list, you can typically:

- View CronJobs in the current project scope
- Open CronJob details
- Create a CronJob
- Create a CronJob from YAML
- Update an existing CronJob
- Delete a CronJob
- Run a CronJob immediately when the UI provides that action

## Create a CronJob

1. Open the CronJob list.
2. Click **Create**.
3. Select the target **Cluster** if required by the page context.
4. Select the target **Namespace**.
5. Configure the schedule, job template, and runtime settings.
6. Click **Create**.

### Common configuration areas

The create or update page can include configuration areas such as:

- Basic information
- Schedule and concurrency policy
- Job history retention
- Job type and retry settings
- Pod configuration
- Container configuration

### Scheduling review points

Before submitting a CronJob, review the following carefully:

- Whether the schedule matches the intended execution window
- Whether concurrent runs are acceptable
- Whether missed or delayed runs could affect downstream tasks
- Whether the Job template includes the required image, command, configuration, and credentials
- Whether the history retention settings are sufficient for troubleshooting

### Schedule and concurrency guidance

When the page exposes scheduling controls, focus on settings such as:

- the cron expression itself
- whether the schedule is interpreted in a cluster-supported timezone or another effective timezone context
- how concurrent executions are handled

Common concurrency behaviors can include:

- allowing overlapping runs
- forbidding a new run while the previous run is still active
- replacing an active run with a newer one

Choose the policy based on whether overlapping execution is safe for the task.

### Job history and retry guidance

Review settings such as:

- how many successful Jobs to retain
- how many failed Jobs to retain
- how many retries are allowed before a run is treated as failed

Use higher history retention when troubleshooting matters more than object cleanup, and lower retention when the schedule is frequent and the generated Job objects would otherwise accumulate quickly.

## Create a CronJob from YAML

1. Open the CronJob list.
2. Click **Create from YAML**.
3. Select the target **Cluster** and **Namespace**.
4. Paste or edit the YAML content.
5. Click **Create**.

## Update a CronJob

1. Open the CronJob list or a CronJob details page.
2. Click **Update**.
3. Modify the supported configuration.
4. Click **Update**.

After updating a CronJob, remember that the new configuration mainly affects future executions rather than Jobs that have already finished.

## Run a CronJob immediately

If the page provides an immediate execution action:

1. Open the CronJob list or details page.
2. Click the action for immediate execution.
3. Confirm the operation.
4. Verify that a corresponding Job is created.

Use this workflow when you need to validate the Job template without waiting for the next scheduled time.

## Operational considerations

When troubleshooting a CronJob, review not only the CronJob object itself, but also the Jobs and Pods created from it.

Typical failure sources include:

- invalid schedules or unexpected effective timezone interpretation
- image pull failures
- missing ConfigMaps, Secrets, or other runtime dependencies
- Job retries exhausted because the container command exits unsuccessfully
- concurrency policy conflicts with long-running executions

## Verify the result

After creating, updating, or executing a CronJob, verify that:

- The CronJob appears in the selected project and cluster context.
- The expected namespace is shown.
- The schedule or execution policy is displayed correctly.
- The next run timing looks reasonable for the intended schedule.
- A Job is created when you trigger immediate execution.
- The resulting Job and Pod complete or fail in the expected way.
- No schedule or runtime errors remain.

After deleting a CronJob, verify that:

- The CronJob no longer appears in the list.
- The deletion outcome for previously created Jobs matches your operational expectations.

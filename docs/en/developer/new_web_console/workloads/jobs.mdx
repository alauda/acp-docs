# Jobs

In the **Applications** view, you can use **Jobs** to track finite workloads that run to completion in the current project and cluster context.

## Understand workload scope

The Job page is opened from the project-scoped **Applications** view, but each Job still runs in a concrete cluster and namespace context.

When you review execution results, verify not only the Job status but also the cluster and namespace where the corresponding Pods were created.

## When to use a Job

A Job is intended for workloads that should run until they complete successfully or stop after failure handling is exhausted.

Use a Job when the work should finish rather than remain running continuously.

Typical scenarios include:

- One-time initialization tasks
- Data import or export tasks
- Batch processing
- Maintenance or recovery operations
- Short-lived execution workflows that should not remain running permanently

If the workload should remain available continuously, use a long-running workload model such as a Deployment instead. If the work should run on a schedule, review [CronJobs](./cronjobs).

## Open the Job list

1. In the **Applications** view, select the target project.
2. In the left navigation pane, click **Jobs**.

From the Job list, you can typically:

- View Jobs in the current project scope
- Open Job details
- Review completion state and runtime information

## Understand Job behavior

Unlike long-running workloads, the main operational question for a Job is usually whether it completed correctly, how long it ran, and what happened when it failed.

### Common execution controls

Depending on how the Job was created, execution behavior can be influenced by settings such as:

- how many successful completions are required
- how many Pods can run in parallel
- how many retries are allowed before the Job is treated as failed
- which restart policy is used for the underlying Pods

When you review a failed Job, confirm whether the failure is from the application logic itself or from exhausted retry behavior after repeated Pod failures.

These controls affect not only whether the Job eventually succeeds, but also how much concurrency, repeated execution, and retry pressure the Job can create in the cluster while it is running.

## View Job details

Open a Job to review its execution result and related runtime information.

Depending on the available integrations, the details area can include status, associated Pods, logs, events, and other execution-related information.

### Review points

When checking a Job, pay attention to:

- Whether the Job completed successfully
- Whether it is still running longer than expected
- Whether repeated failures or retries occurred
- Whether the created Pods show image, scheduling, permission, or configuration issues
- Whether the completion behavior matches the intended task design

### Execution interpretation guidance

When you evaluate Job outcome, focus especially on:

- whether the exit result reflects application success or only temporary retry progress
- whether parallelism and completion expectations match the actual number of created Pods
- whether restart policy or retry settings are masking an underlying workload problem
- whether a failed Job should be rerun only after its configuration or dependencies are corrected

## Troubleshoot common Job issues

If a Job does not complete as expected, check the following first:

- the Pod exit code and container logs
- image pull failures or missing registry credentials
- missing ConfigMaps, Secrets, PVCs, or runtime permissions
- invalid command or argument settings
- retry exhaustion after repeated Pod failure

When available in the UI, use the Job details, Pod logs, and related events together rather than relying on the Job phase alone.

## Post-completion review

After a Job finishes, keep the related Pods and logs available long enough for the level of verification or troubleshooting you still need.

A completed Job can still be operationally important if its logs, output behavior, or failure history are needed to judge whether the task really succeeded.

## Verify the result

When reviewing a Job, verify that:

- The Job appears in the selected project and cluster context.
- The expected namespace is shown.
- The completion status matches the expected execution result.
- Related Pods and logs are available when needed.
- The available runtime information is sufficient to judge success or failure.
- The observed Pod count, retry behavior, and execution duration are consistent with the intended Job design.

# Secrets

Use this page when you need to manage Secrets from the **Clusters** view.

In this workflow, you enter from a concrete cluster. Each Secret still belongs to a specific namespace, so detail, update, and workload-consumption behavior remain namespace-scoped even though the surrounding entry point is cluster-oriented.

## Before you begin

Make sure that:

- You have opened the target cluster in the **Clusters** view.
- You know which namespace should contain the Secret.
- You understand the Secret type and the workloads that will reference it.
- You understand the operational impact if workloads, image pulls, or mounted files depend on this Secret.

Use Secrets for credentials, keys, certificates, tokens, or image-pull authentication.

## Open the Secrets page

1. In the ACP console, go to **Clusters**.
2. Open the target cluster.
3. Open the Secrets page for the target namespace context exposed by the console.

## What you can do from this page

Typically, you can:

- Create a Secret
- View Secret details
- Update a Secret
- Review YAML or structured data
- Delete a Secret

## Create a Secret

1. Open the Secrets page in the target cluster context.
2. Click **Create**.
3. Select or confirm the target namespace if required.
4. Choose the correct Secret type.
5. Enter the required data.
6. Submit the form.

### Common Secret types

Depending on the current UI behavior, the page can support Secret types such as:

- **Opaque** for general key-value sensitive data
- **Basic authentication** for username and password pairs
- **TLS** for certificates and private keys
- **SSH authentication** for SSH private keys or related authentication data
- **Image pull** credentials for private registries

Depending on the edit mode, the console can help convert the entered data into the Kubernetes Secret representation used in YAML.

A workload can only reference Secrets from the same namespace.

## How workloads typically consume Secrets

Secrets are commonly consumed in workloads through patterns such as:

- Injecting a Secret key as an environment variable
- Injecting all supported keys as environment variables
- Mounting Secret data as files in a volume
- Using registry credentials through image-pull configuration

Before you submit the Secret, confirm that the target workload expects the selected Secret type and that the key names match the consuming configuration.

## Update or delete a Secret

1. Open the Secrets list.
2. Open the target Secret.
3. Use the available update or delete action.
4. Confirm the change.

:::warning
If a workload references a Secret, updating or deleting that Secret can affect workload startup, image pulling, mounted files, or environment-variable injection. Some changes require a rollout, restart, or Pod recreation before the updated configuration takes effect.
:::

Do not delete platform-generated or system-critical Secrets unless you explicitly understand the impact. This caution especially applies to authentication-related or service-account-related Secrets that can be required for platform behavior.

## Verify the result

After creating or updating a Secret, verify that:

- The Secret appears in the expected namespace.
- The Secret type and key set are correct.
- Referencing workloads can still authenticate, mount, or pull images as expected.
- Any required rollout, restart, Pod recreation, or image-pull revalidation has been completed.

After deleting a Secret, verify that:

- The Secret no longer appears in the list.
- No workload restart, recreate, mount, or image pull now fails because the referenced Secret is missing.

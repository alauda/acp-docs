# ConfigMaps

Use this page when you need to manage ConfigMaps from the **Clusters** view.

In this workflow, you enter from a concrete cluster. Each ConfigMap still belongs to a specific namespace, so create, update, and detail operations remain namespace-scoped even though the surrounding entry point is cluster-oriented.

## Before you begin

Make sure that:

- You have opened the target cluster in the **Clusters** view.
- You know which namespace should contain the ConfigMap.
- The configuration data does not contain credentials or other sensitive material.
- You understand which workloads in that namespace will consume the configuration.

If the data is sensitive, use [Secrets](./secrets) instead.

## Open the ConfigMaps page

1. In the ACP console, go to **Clusters**.
2. Open the target cluster.
3. Open the ConfigMaps page for the target namespace context exposed by the console.

## What you can do from this page

Typically, you can:

- Create a ConfigMap
- View ConfigMap details
- Update a ConfigMap
- Review YAML or key-value content
- Delete a ConfigMap

## Create a ConfigMap

1. Open the ConfigMaps page in the target cluster context.
2. Click **Create**.
3. Select or confirm the target namespace if required.
4. Enter the ConfigMap name.
5. Configure the entries.
6. Submit the form.

### Common entry patterns

Depending on the current UI behavior, the create or update page can support operations such as:

- Adding key-value entries one by one
- Bulk pasting multiple `key=value` lines
- Importing text files as entries
- Importing binary files where supported by the UI

When you use bulk paste, keep one `key=value` pair on each line. For example:

```text
APP_ENV=production
LOG_LEVEL=info
FEATURE_FLAG=true
```

A workload can only reference ConfigMaps from the same namespace.

## How workloads typically consume ConfigMaps

ConfigMaps are commonly consumed in workloads through patterns such as:

- Injecting all entries as environment variables
- Injecting a specific key as a single environment variable
- Mounting keys as files in a volume
- Reusing the same ConfigMap across multiple workloads in the same namespace

Before you submit the ConfigMap, confirm that the keys, expected file names, and value formats match the way the workload reads them.

## Update or delete a ConfigMap

1. Open the ConfigMaps list.
2. Open the target ConfigMap.
3. Use the available update or delete action.
4. Confirm the change.

:::warning
If a workload references a ConfigMap through environment variables or mounted files, updating or deleting that ConfigMap can affect workload behavior. Some changes require a rollout, restart, or Pod recreation before the updated configuration takes effect.
:::

Depending on the supported entry type, imported binary content can be more restrictive to edit after creation than regular text entries. Confirm the current UI behavior before using ConfigMaps for frequently changing binary content.

## Verify the result

After creating or updating a ConfigMap, verify that:

- The ConfigMap appears in the expected namespace.
- The updated data is displayed correctly.
- The value format matches how the workload reads the configuration.
- Referencing workloads can still read the intended configuration.
- Any required rollout, restart, or Pod recreation has been completed.

After deleting a ConfigMap, verify that:

- The ConfigMap no longer appears in the list.
- No workload restart, recreate, or rollout now fails because the referenced ConfigMap is missing.

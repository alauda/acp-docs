# Namespaces

Use this page when you need to manage namespaces from the **Clusters** view.

This workflow is different from the project-scoped namespace workflow in the **Applications** view. In the **Clusters** view, you enter from a concrete cluster and operate on namespaces as cluster resources.

## Before you begin

Make sure that:

- You have opened the target cluster in the **Clusters** view.
- You understand whether you are creating a new namespace, updating an existing namespace, or reviewing namespace-level policies and limits.
- You have the required permissions for namespace management in the target cluster.

## Understand cluster-scoped namespace management

In the **Clusters** view, namespace management is cluster-oriented rather than project-oriented.

In practice, this means:

- You start from a concrete cluster instead of a project workspace.
- You manage the namespace as a cluster resource.
- This page does not describe project-association workflows such as importing a namespace into a project or removing it from a project.

Use this entry when the main question is cluster-side namespace governance, visibility, or lifecycle management rather than project membership.

## Open the namespace list

1. In the ACP console, go to **Clusters**.
2. Open the target cluster.
3. Find and open the **Namespaces** page.

## What you can do from this page

Typically, you can:

- Create a namespace
- Open namespace details
- Update namespace settings
- Delete a namespace
- Review namespace YAML
- Review resource quotas
- Review limit ranges

## Create a namespace

1. Open the **Namespaces** list in the target cluster.
2. Click **Create Namespace**.
3. Enter the required namespace information.
4. Submit the form.

### Create review points

Before creating the namespace, confirm:

- that the name follows your cluster naming expectations
- that any labels or annotations reflect the intended operational use
- that the namespace-level quota, limit, or policy settings match the workloads that will run there

Labels, annotations, quotas, and limit settings should be treated as workload-admission and resource-governance inputs, not only as descriptive metadata. Small policy differences here can change how later workloads are admitted, scheduled, or defaulted.

## Update a namespace

1. Open the **Namespaces** list.
2. Find the target namespace.
3. Use the action menu or the available update entry.
4. Modify the required settings.
5. Save the change.

When supported by the current UI, updates can include quota, limit-range, metadata, or other namespace-level governance settings.

A namespace update often affects future workload behavior more than existing running Pods immediately. Review the change in terms of what it will allow, deny, or default for later workload creation, rollout, or scaling.

## View namespace details

1. Open the target namespace.
2. Review the available tabs or sections.

Depending on the current environment, the details page can expose information such as:

- Basic namespace information
- YAML
- Resource quotas
- Limit ranges

## Operational considerations

Namespace-level policy changes can affect later Pod admission, scheduling, and resource consumption.

For example:

- stricter admission policies can block Pod creation
- tighter quotas can block later workload scaling or creation
- stricter limit settings can change default requests or limits for newly created workloads

When you review namespace governance settings, think about both current workloads and the next workload change that a team might attempt in the same namespace.

## Delete a namespace

1. Open the target namespace action menu from the namespace list, or use the delete action if it is exposed on the details page.
2. Confirm the deletion.

Be careful when deleting a namespace because it can remove or affect the resources running inside that namespace.

Namespace deletion is usually a broad cleanup operation rather than a narrow configuration change. Before confirming deletion, verify that workloads, configuration resources, storage usage, and any remaining troubleshooting needs are no longer required in that namespace.

## Verify the result

After the operation:

- Confirm that the namespace appears in the namespace list if you created it.
- Confirm that updated information is shown correctly if you changed namespace settings.
- Confirm that the namespace details page, YAML, quota, and limit information are accessible when expected.
- Confirm that the namespace no longer appears in the list if you deleted it.
- Confirm that the resulting workload and resource state in the cluster matches your intended outcome.
- Confirm that later workload admission, scaling, or governance behavior matches the updated namespace policy settings when those were part of the change.

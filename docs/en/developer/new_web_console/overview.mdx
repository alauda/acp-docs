# Overview

Use the web console documentation according to the task you need to complete, not only according to the page you are currently viewing.

This section is most useful when you need to understand how ACP organizes user-facing console workflows and where to find the right operational guide.

## Start with the right task area

Choose the entry point that best matches your current goal.

### Work with project-scoped applications

Use the **Applications** view guidance when you need to:

- Review application health and project-scoped runtime information
- Create or update applications in the selected project and cluster context
- Manage namespaces used by application workloads
- Work with workloads and application configuration resources

### Understand the web console section structure

The migrated web console docs in this area currently focus on the operational flows exposed through the **Applications** view.

Some of the same Kubernetes resource types can also be available from the **Clusters** view. When that happens, treat the two experiences as separate documentation targets rather than combining both entry paths and scopes into one workflow page.

The web console can also expose other views for platform-level or project-level administration depending on the current environment, enabled features, and permissions.

In practice, this means:

- Top-level hub pages help you choose a task family.
- Topic pages explain the user-visible workflow in the current console.
- Deeper conceptual guidance remains in other ACP documentation areas and is linked when that content is already authoritative.

## How to use these docs effectively

When you open a page in this section, use it in this order:

1. Confirm that you are in the correct project, cluster, and namespace context.
2. Follow the entry steps to open the relevant list or details page.
3. Complete the target task.
4. Use the **Verify the result** section before assuming the operation succeeded.
5. Open the linked conceptual or workload-specific docs if you need deeper parameter guidance.

## Understand scope and variability

Many web console pages are context-sensitive.

In this module, the main interaction model is project-scoped application management with cluster and namespace context.

In practice, this means:

- You usually enter a page from the current project scope in the **Applications** view.
- Lists and dashboards can summarize data across the clusters associated with the current project.
- When you create, update, or inspect a specific application, workload, ConfigMap, or Secret, the operation is still resolved in a concrete cluster and namespace context.
- Configuration and workload relationships remain namespace-bound even when the surrounding view is project-scoped.

Depending on the current project, enabled integrations, and your permissions, a page can expose different tabs, actions, warnings, or monitoring data.

Because of that, the documentation in this section focuses on:

- Stable user workflows
- Common operational decisions
- Verification steps
- Links to deeper authoritative references where exact field behavior matters

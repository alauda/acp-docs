# Overview

Use this page to understand how the transitional `developer/new_web_console` path relates to the consolidated `developer` and `building_application` pages.

This page is most useful when you need scope guidance, entry-path differences, or a quick decision on which long-term page should host your task.

## Prefer the consolidated destination

In most cases, start from [Building Applications](../building_application) instead of staying in `developer/new_web_console`.

- Use `create_applications` for application creation and update tasks.
- Use `application_workloads` for workload-specific runtime operations.
- Use `namespace` for namespace preparation and governance.
- Use `configuration` for ConfigMaps and Secrets.
- Use `operation_maintaining` and `application_observability` for application operations, monitoring, logs, events, versions, and related day-2 work.

## Keep this path for transition and scope notes

Keep using `developer/new_web_console` only when one of these conditions applies:

- You entered from an existing page path under `developer/new_web_console`.
- You need to compare **Applications** view and **Clusters** view behavior before the remaining cluster-oriented pages are fully absorbed.
- You need short-term routing guidance while older links are still being retired.

## Understand scope and variability

Many web console pages are context-sensitive.

The main interaction model in this transitional area is still project-scoped application management with cluster and namespace context.

In practice, this means:

- Lists and dashboards can summarize data across the clusters associated with the current project.
- Specific create, update, inspect, or troubleshoot actions still resolve in a concrete cluster and namespace context.
- The same resource type can appear under **Applications** view and **Clusters** view, but the entry path and task framing can still differ.

Depending on the current project, enabled integrations, and your permissions, a page can expose different tabs, actions, warnings, or monitoring data.

Because of that, this area now focuses on entry scope, routing, and remaining view-specific differences rather than repeating all downstream task navigation.

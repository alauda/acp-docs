# Creating applications from Code

Creating application from code is implemented using Source to Image(S2I) technology. S2I is an automated framework for building container images directly from source code. This approach standardizes and automates the application build process, allowing developers to focus on source code development without worrying about containerization details.

## Prerequisites
- Complete the installation of [Alauda Container Platform Builds](../../s2i/install_builds/install_builds_operator.mdx)

## Procedure

1. **Container Platform**, and navigate to **Application** > **Applications**.

2. Click **Create**.

3. Select the **Create from Code**.

4. For detailed parameter descriptions, please refer to [Managing applications created from Code](/developer/s2i/functions/s2i_application_management.mdx#procedure)

5. After completing the parameter input, click **Create**.

6. The corresponding deployment can be viewed on the **Detail Information** page.

## Create applications from Git in the Applications view

If the current environment exposes Git-based creation in the **Applications** view, you can combine source retrieval, image build settings, workload settings, and optional network exposure in a single workflow.

1. Open the target application page in the **Applications** view.
2. Click **Create from Git**.
3. Select the target **Cluster**.
4. Configure the application metadata, repository settings, build settings, deployment settings, and optional network exposure.
5. Review the configuration and optionally enable **Build Immediately**.
6. Click **Create**.

If the selected environment does not support the required build capability, the workflow can display an error state and cannot continue normally.

## Review repository and build settings

When you configure the repository section, review:

- Repository URL
- Optional revision
- Optional source subdirectory
- Optional credential Secret for repository access

The repository validation result helps confirm whether the selected source tree and access method match the intended build. A validation warning does not always block the workflow, but it can indicate that the detected repository structure, revision, or access method still needs review.

In the build section, confirm that the selected build image and build method match the repository technology and expected output. The build image choice affects not only image generation but also compatibility with the repository language, build process, and target runtime behavior.

## Review deployment and network settings

After the repository and build settings are confirmed, review the workload configuration together with optional service exposure.

Pay special attention to:

- replica count
- resource requests and limits
- environment variables and referenced configuration
- probes and Pod-level runtime settings
- target port and service exposure settings
- ingress-related configuration when external access is enabled

Successful source retrieval and image build do not guarantee a healthy runtime workload. Referenced ConfigMaps, Secrets, probes, ports, and resource settings still determine whether the application starts correctly after the image is built.

## Review points before submission

Before clicking **Create**, confirm that:

- The selected cluster supports the required Git build capability.
- The repository URL, revision, and optional subdirectory identify the intended source tree.
- Any required credential Secret is valid in the selected namespace.
- The selected build image and build method match the repository technology.
- Referenced ConfigMaps, Secrets, and runtime dependencies already exist when the workload expects them.
- The selected port and exposure settings match how the application actually serves traffic.
- The requested resources and health checks are realistic for the first startup of the built application.

## Verify the result

After the creation completes, verify that:

- The application appears in the expected project, cluster, and namespace context.
- The build starts as expected if **Build Immediately** is enabled.
- The build completes successfully when the source and build configuration are valid.
- The created workload, service, and ingress resources are visible when configured.
- The resulting Pods start successfully and the runtime behavior matches the selected workload and exposure settings.

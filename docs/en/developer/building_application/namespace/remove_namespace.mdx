# Deleting/Removing Namespaces

You can either delete a namespace permanently or remove it from the current project.

## Deleting Namespaces

**Delete Namespace**: Permanently deletes a namespace and all resources within it (e.g., Pods, Services, ConfigMaps). This action cannot be undone and releases allocated resource quotas.

```bash
kubectl delete namespace <namespace-name>
```

## Removing Namespaces

**Remove Namespace**: Removing a namespace from the current project without deleting its resources. The namespace remains in the cluster and can be imported into other projects via [Import Namespace](./import_namespace.mdx).

:::note
- This feature is exclusive to the <Term name="product"/>.
- Kubernetes does not natively support "removing" namespaces from projects.
:::

```bash
kubectl label namespace <namespace-name> cpaas.io/project- --overwrite
```

## How to choose between remove and delete

Use **Remove Namespace** when the namespace resource should remain in the cluster but should no longer belong to the current project.

Use **Delete Namespace** when the namespace resource itself is no longer needed and the workloads and namespaced resources inside it can be removed.

Before deleting a namespace, verify that workloads, ConfigMaps, Secrets, storage usage, and any remaining troubleshooting needs are no longer required in that namespace.

## Removing or deleting from the Applications view

In the **Applications** view of the new web console, distinguish clearly between project disassociation and resource deletion:

- **Remove Namespace** changes project association and governance visibility.
- **Delete Namespace** deletes the namespace resource from the cluster.

After either action, verify that the resulting project association or cluster state matches your actual intent and that no dependent application workflow is now blocked.

## Deleting from the Clusters view

In the **Clusters** view of the new web console, namespace management is cluster-oriented, so deletion is the main lifecycle removal action.

Treat deletion there as broad cleanup of a cluster resource rather than a narrow configuration change. Confirm that the resulting workload and resource state in the cluster matches the intended outcome.

## Removing or deleting a namespace in the new web console

This page mainly describes the legacy console workflow, but namespace removal or deletion in the new web console can start from different views.

### Remove from the Applications view

Use the **Applications** view when you need to remove the namespace from the current project without necessarily deleting the namespace resource itself.

Before confirming the action, verify that:

- you are removing project association rather than deleting the namespace resource
- the resulting project membership still matches the intended governance model
- no dependent application workflow is blocked because the namespace is no longer associated with the project

### Delete from the Clusters view

Use the **Clusters** view when you need to delete the namespace resource from a concrete cluster.

Before confirming the action, verify that:

- workloads, ConfigMaps, Secrets, storage usage, and troubleshooting data in that namespace are no longer required
- the deletion is intended as a cluster cleanup action rather than a project-association change
- the resulting cluster resource state matches the intended operational outcome

# Version Management for Applications

After updating the application through the platform interface, a historical version record is automatically generated. For application updates triggered by non-interface operations, such as updating the application via API calls, you can manually create a version snapshot to record the changes.

**Note**: When the number of version snapshot entries exceeds 6, the platform retains only the latest 6 entries and automatically deletes the others, prioritizing the removal of the oldest version snapshot entries.

Use version management when you need to review what changed, preserve a known-good state before a risky update, compare the current definition with an earlier revision, or recover to a previous configuration.

The history list can help you determine whether an entry represents:

- a normal version progression
- a manually created snapshot
- a rollback-derived state
- a version with meaningful configuration differences from the current state

## Creating a Version Snapshot

### Procedure

1. Access **Container Platform**.

2. In the left navigation bar, click **Application Management** > **Native Applications**.

3. Click on ***Application Name***.

4. In the **Version Snapshot** tab, click **Create Version Snapshot**.

5. Configure the information and click **Confirm**.

   **Note**: You can also [Distribute the Application](./export.mdx), which allows you to distribute the version snapshot of the application as a Chart, facilitating the rapid deployment of the same application across multiple clusters and namespaces on the platform.

### Review points before creating a snapshot

Before creating a snapshot, confirm that:

- The current application state is the one you want to preserve.
- Related ConfigMaps, Secrets, images, and service relationships already reflect the intended configuration.
- You are not preserving a known broken runtime state by mistake.

## Rolling Back to a Historical Version

Roll back the current application's configuration to a historical version.

### Procedure

1. Access **Container Platform**.

2. In the left navigation bar, click **Application Management** > **Native Applications**.

3. Click on ***Application Name***.

4. In the **Historical Versions** tab, click on ***Version Number***.

5. Click ⋮ > **Roll Back to This Version**.

6. Click **Roll Back**.

### Review points before rollback

Before rollback, confirm that:

- The selected version is the intended recovery point.
- External dependencies referenced by that version still exist and remain compatible.
- The rollback will not reintroduce a configuration that is obsolete for the current environment.
- The expected impact on traffic, data handling, or workload state is acceptable.

### Comparing historical versions

When reviewing a historical version, pay special attention to differences in:

- image or tag selection
- replica or workload settings
- ConfigMap and Secret references
- network exposure resources
- associated resource membership

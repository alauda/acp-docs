# Jobs

In the **Applications** view, you can use **Jobs** to track finite workloads that run to completion in the current project and cluster context.

## Understanding Jobs

Refer to the official Kubernetes documentation: [Jobs](https://kubernetes.io/docs/concepts/workloads/controllers/job/)

A **Job** provide different ways to define tasks that run to completion and then stop. You can use a Job to define a task that runs to completion, just once.

- **Atomic Execution Unit**: Each Job manages one or more Pods until successful completion.
- **Retry Mechanism**: Controlled by `spec.backoffLimit` (default: 6).
- **Completion Tracking**: Use `spec.completions` to define required success count.

## YAML file example

```yaml
# example-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: data-processing-job
spec:
  completions: 1 # Number of successful completions required
  parallelism: 1 # Maximum parallel Pods
  backoffLimit: 3 # Maximum retry attempts
  template:
    spec:
      restartPolicy: Never # Job-specific policy (Never/OnFailure)
      containers:
        - name: processor
          image: alpine:3.14
          command: ['/bin/sh', '-c']
          args:
            - echo "Processing data..."; sleep 30; echo "Job completed"
```

## Execution Overview

Each Job execution in Kubernetes creates a dedicated Job object, enabling users to:

- **Creating a job via**

  ```bash
  kubectl apply -f example-job.yaml
  ```

- **Track job lifecycle via**

  ```bash
  kubectl get jobs
  ```

- **Inspect execution details via**

  ```bash
  kubectl describe job/<job-name>
  ```

- **View Pod logs via**

  ```bash
  kubectl logs <pod-name>
  ```

## Understand workload scope

The Job page is opened from the project-scoped **Applications** view, but each Job still runs in a concrete cluster and namespace context.

When you review execution results, verify not only the Job status but also the cluster and namespace where the corresponding Pods were created.

## When to use a Job

Use a Job when the work should finish rather than remain running continuously.

Typical scenarios include:

- One-time initialization tasks
- Data import or export tasks
- Batch processing
- Maintenance or recovery operations
- Short-lived execution workflows that should not remain running permanently

If the workload should remain available continuously, use a long-running workload model such as a Deployment instead. If the work should run on a schedule, review [CronJobs](./cronjob_create.mdx).

## Review points for Job execution

When checking a Job, pay attention to:

- Whether the Job completed successfully
- Whether it is still running longer than expected
- Whether repeated failures or retries occurred
- Whether the created Pods show image, scheduling, permission, or configuration issues
- Whether the completion behavior matches the intended task design

## Execution interpretation guidance

When you evaluate Job outcome, focus especially on:

- whether the exit result reflects application success or only temporary retry progress
- whether parallelism and completion expectations match the actual number of created Pods
- whether restart policy or retry settings are masking an underlying workload problem
- whether a failed Job should be rerun only after its configuration or dependencies are corrected

## Post-completion review

After a Job finishes, keep the related Pods and logs available long enough for the level of verification or troubleshooting you still need.

A completed Job can still be operationally important if its logs, output behavior, or failure history are needed to judge whether the task really succeeded.

## Verify the result

When reviewing a Job, verify that:

- The Job appears in the expected project, cluster, and namespace context
- The completion status matches the expected execution result
- Related Pods and logs are available when needed
- The available runtime information is sufficient to judge success or failure
- The observed Pod count, retry behavior, and execution duration are consistent with the intended Job design

# \<Title: Generally the Name of the Problem>

## Problem Description

Detail the environment in which the problem occurs, its scope of impact, and the symptoms, while appropriately adding the following supplementary information:

- Problem Screenshot: Currently, it is preferred to use textual descriptions. If words cannot sufficiently describe the issue, screenshots should be used, and users are required to register and discuss this with the documentation team.
- Relevant Logs: Include error messages, system logs, debugging information, etc., in the form of code blocks.

## Root Cause Analysis

List one or more actual causes that have occurred and led to this problem (this can be a single confirmed cause or multiple possible causes). For each cause, the following conditions must be met:

1. It must be an actual reason that has occurred and has been confirmed to lead to the problem, rather than speculative or imagined causes.
2. Detailed cause-and-effect analysis must be provided, including:
   - How this cause triggered the problem.
   - The specific mechanism and process by which the problem occurred.

## Problem Diagnosis (Optional)

List the specific steps for diagnosing the problem in detail. Through problem diagnosis, users can identify the most likely or most certain cause among multiple reasons and use appropriate solutions to resolve the issue. When diagnosing problems, it should be noted:

- If there is only one root cause, then diagnosis steps must be provided. In this case, it is recommended to present them in an ordered list format.

  ```
  1. Diagnosis Step 1
  ...
  n. Diagnosis Step N
  ```

- If there are multiple root causes and the diagnosis steps are different, then each root cause should be listed under a third-level heading, and detailed diagnosis steps should be provided under each cause.
  ```
  ### Root Cause 1
    1. Diagnosis Step 1
    ...
    n. Diagnosis Step N
  ...
  ### Root Cause N
    1. Diagnosis Step 1
    ...
    n. Diagnosis Step N
  ```

## Solution for Root Cause 1 (This is the heading format for multiple causes. If there is a single root cause, the name of this second-level heading should be "Solution.")

Specific precautions, prerequisites, and operational steps to resolve the issue.

> **Note**: If there is only one root cause, the name of the second-level heading should be "Solution"; if there are multiple root causes and the solutions differ for each, then each solution should be presented under a separate second-level heading with names such as "Solution for Root Cause 1", "Solution for Root Cause N", etc.

### Precautions (Optional)

Precautions to consider before starting operations. Failing to read this carefully may result in operational failure or even more serious consequences.

### Prerequisites (Optional)

Conditions that must be met before proceeding with the operational steps.

### Operational Steps (This third-level heading can be omitted if there are no precautions or prerequisites)

- Operational steps are generally presented in an ordered list format; if the steps are complicated, additional headings can be added to distinguish between different steps. In this case, fourth-level headings can be used to list each step, with each step detailed using an ordered list format.
- If the operations are conducted on a container platform, all steps can be listed in detail. If operating on a third-party platform or other software, only provide the key steps.

  > Reason: For non-container platform operations, being overly detailed may lead to documentation issues when the interface changes, which can cause many problems. To ensure the long-term usability of the documentation, descriptions of third-party platform operational steps should follow these principles:
  >
  > - Highlight core steps and key operations.
  > - Avoid excessive reliance on specific interface details.
  > - Maintain the generality and flexibility of the operational step descriptions.
  > - Ensure users understand the purpose of the operations and the basic flow.

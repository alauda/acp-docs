# \<Title: Usually the problem name>

## Problem Description

Provide a detailed account of the environment in which the problem occurs, its impact range, and the symptoms of the problem, while also including the following auxiliary information as appropriate:

- Problem Screenshot: Currently, it is recommended to use text descriptions. If text cannot adequately describe the issue, screenshots should be used, and users need to register and discuss with the documentation team.
- Related Logs: Use code blocks to include error messages, system logs, debug information, etc.

## Root Cause Analysis

List one or more actual causes that have occurred and led to this problem (these can be single confirmed causes or multiple possible causes). Each cause should meet the following conditions:

1. It must be a real, confirmed cause that has occurred and proven to lead to the problem, rather than a speculative or imagined reason.
2. Detailed causal analysis should be provided, including:
   - How this cause triggers the problem.
   - The specific mechanisms and processes through which the problem occurs.

## Problem Diagnosis (Optional)

List specific steps to diagnose the problem in detail. Through problem diagnosis, users can identify the most likely or certain cause among multiple reasons and apply the appropriate solution to resolve the issue. Attention should be paid to the following during the diagnosis:

- If there is only one root cause, then diagnostic steps need to be provided. It is recommended to display these in an ordered list format.

  ```
  1. Diagnostic Step 1
  ...
  n. Diagnostic Step N
  ```

- If there are multiple root causes with different diagnostic steps, each root cause should be listed under a third-level heading, and detailed diagnostic steps must be provided under each cause.
  
  ```
  ### Root Cause 1
    1. Diagnostic Step 1
    ...
    n. Diagnostic Step N
  ...
  ### Root Cause N
    1. Diagnostic Step 1
    ...
    n. Diagnostic Step N
  ```

## Solution for Root Cause 1 (This is the title format for multiple root causes; if there is a single root cause, the second-level title should be "Solution.")

Specific notes, prerequisites, and operational steps for solving the problem.

> **Note**: If there is only one root cause, the second-level title should be "Solution." If there are multiple root causes, and each has a different solution, each solution should have its own second-level title, named "Solution for Root Cause 1," "Solution for Root Cause N," etc.

### Notes (Optional)

Items that need to be noted before beginning operations. Failing to read this section carefully may lead to operational failures or even more severe consequences.

### Prerequisites (Optional)

Conditions that must be met before starting the operational steps.

### Operational Steps (These subheadings can be omitted if there are no notes or prerequisites)

- Operational steps are generally displayed in an ordered list format. If the steps are complex, additional headings may be needed to differentiate between multiple steps. In this case, a fourth-level heading can be used to list each step, with specific sub-steps presented in an ordered list format.
- If the operations are carried out on a container platform, each operation step should be detailed. If conducted on a third-party platform or other software, only the key steps should be provided.

  > Reason: Non-container platform operations written too conclusively may lead to issues when the interface changes, thus we cannot guarantee timely updates to the document. To ensure the long-term usability of the document, the operational steps for third-party platforms should adhere to the following principles:
  >
  > - Highlight core steps and key operations.
  > - Avoid excessive reliance on specific interface details.
  > - Maintain general applicability and flexibility in operational step descriptions.
  > - Ensure users understand the purpose and basic process of the operations.

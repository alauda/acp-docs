# \<Title: Generally the Name of the Issue>

## Issue Description

Provide detailed information about the environment in which the issue occurs, its impact scope, and the phenomena of the issue, and include the following supplementary information as appropriate:

- Problem Screenshots: Currently, it is recommended to use text descriptions first. If the text cannot adequately describe the issue, use screenshots. Users need to register and discuss with the documentation team.
- Related Logs: Use code blocks to include error messages, system logs, debugging information, etc.

## Root Cause Analysis

List one or more actual causes that have been confirmed to lead to this issue (it can be a single determined cause or multiple possible causes). Each cause must meet the following criteria:

1. It must be a real cause that has occurred and is verified to lead to this issue, rather than a speculative or imagined reason.
2. Detailed causal analysis must be provided, including:
   - How this cause triggers the issue.
   - The specific mechanisms and processes involved in the occurrence of the issue.

## Issue Troubleshooting (Optional)

List the specific steps to troubleshoot the issue in detail. Troubleshooting allows users to select the most likely or certain cause from multiple causes and apply the appropriate solution to resolve the issue. When troubleshooting, keep in mind:

- If there is only one root cause, you need to provide troubleshooting steps, preferably presented in an ordered list format.

  ```
  1. Troubleshooting Step 1
  ...
  n. Troubleshooting Step N
  ```

- If there are multiple root causes with different troubleshooting steps, use a third-level heading format for each root cause, and provide detailed troubleshooting steps under each root cause.
  ```
  ### Root Cause 1
    1. Troubleshooting Step 1
    ...
    n. Troubleshooting Step N
  ...
  ### Root Cause N
    1. Troubleshooting Step 1
    ...
    n. Troubleshooting Step N
  ```

## Solution for Root Cause 1 (This format applies to multiple root causes; if there is a single root cause, the title of this second-level header should be "Solution")

Specific precautions, prerequisites, and operational steps for resolving this issue.

> **Note**: If there is only one root cause, the title of the second-level header should be "Solution"; if there are multiple root causes with different solutions, each solution should be separately titled "Solution for Root Cause 1," "Solution for Root Cause N," and so on.

### Precautions (Optional)

Important considerations to note before starting operations. If this section is not carefully read, it may lead to operational failures or even more serious consequences.

### Prerequisites (Optional)

The conditions that must be met before starting the operational steps.

### Operational Steps (This third-level header can be omitted if there are no precautions or prerequisites)

- Operational steps are generally presented in an ordered list format. If the operational steps are quite complex, additional titles can be added between steps for differentiation. In this case, a fourth-level heading can be used to list each step, with specific steps detailed in an ordered list format for each step.
- If operations are conducted in a container platform, list each detailed step of the operation completely; if operations are done on a third-party platform or other software, only give the corresponding key steps.

  > Reason: For operations not on a container platform, if written too detailed, we cannot ensure the documentation is updated in a timely manner when its interface changes, which could cause several issues. To ensure the long-term availability of the documentation, the description of operational steps on third-party platforms should follow these principles:
  >
  > - Highlight core steps and key actions.
  > - Avoid over-reliance on specific interface details.
  > - Maintain the generality and flexibility of the operational steps description.
  > - Ensure users understand the purpose and basic process of the operation.

# Создание задачи инспекции

Задача инспекции представляет собой планирование и управление инспекциями для конкретных экземпляров компонентов. При создании задачи инспекции вы можете указать экземпляры компонентов, которые необходимо инспектировать, а также цикл и стратегию инспекции. Вы также можете указать максимальное количество записей инспекции для сохранения; любые результаты инспекции, превышающие это количество, будут автоматически перезаписаны.

## Процедура

<Tabs>
  <Tab label="CLI">
    При создании задачи инспекции вы можете использовать следующую команду для её создания:

    ```bash
    $ kubectl -n <namespace> apply -f - <<EOF
    {
      "apiVersion": "middleware.alauda.io/v1",
      "kind": "Inspection",
      "metadata": {
        "name": "insp-task",
        "namespace": "tongrds-1"
      },
      "spec": {
        "component": "mysqlcluster",
        "instances": [
          {
            "name": "mgr1"
          }
        ],
        "jobsHistoryLimit": 3,
        "trigger": "Manual"
      }
    }
    EOF
    ```

    ::: info

    - `<name>` — это имя ресурса задачи инспекции; вы можете установить имя, которое легко запомнить.
    - `<namespace>` — это пространство имен, к которому принадлежит задача инспекции, а также целевое пространство имен для инспекции. Например, если вы хотите инспектировать экземпляр компонента `mgr1` в пространстве имен `tongrds-1`, вы должны установить `<namespace>` на `tongrds-1`.
    - `<spec.component>` используется для указания целевого типа компонента для задачи инспекции, такого как `mysqlcluster`.
    - `<spec.instances.name>` используется для указания целевого имени экземпляра для задачи инспекции; вы можете указать несколько экземпляров в одном пространстве имен или использовать подстановочный знак `*`, чтобы соответствовать всем экземплярам.
    - `<spec.jobsHistoryLimit>` используется для указания максимального количества записей инспекции, которые текущая задача инспекции будет сохранять. Любые записи, превышающие это значение, будут автоматически перезаписаны.
    - `<spec.trigger>` используется для указания метода запуска текущей задачи инспекции, в настоящее время поддерживает `Manual` и `Schedule`, которые представляют собой ручной запуск и запланированный запуск соответственно. Если установлено значение `Schedule`, также должен быть установлен следующий параметр:
      - `<spec.schedule>` используется для указания времени запуска текущей задачи инспекции, например, `0 0 1 * *` указывает на запуск в 1 час ночи каждый день. Для получения дополнительной информации о запланированных задачах, пожалуйста, обратитесь к [Политикам запланированной инспекции](../how_to/inspection_policy.mdx).

    :::
  </Tab>

  <Tab label="Web Console">
    1. Войдите в раздел `Задачи` инспекции

    2. Выберите проекты, которые вы хотите инспектировать, в верхнем левом углу.

    3. Нажмите кнопку `Создать задачи` и заполните информацию, требуемую в всплывающей форме.

    ![Форма создания инспекции](../../../../../en/appservice/unified-om/inspection-mgt/functions/assets/create_inspection_form.png)

    4. После заполнения вышеуказанной формы и нажатия на `Создать`, задача инспекции сразу появится в списке задач инспекции.
  </Tab>
</Tabs>

После успешного создания задачи инспекции служба на стороне сервера выполнит соответствующие операции инспекции в соответствии с настройками задачи инспекции. Для получения дополнительной информации о выполнении инспекций, пожалуйста, обратитесь к [Выполнение задач инспекции](./run_inspection_task.mdx).

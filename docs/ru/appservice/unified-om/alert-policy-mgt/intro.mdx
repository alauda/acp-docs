# Введение

В процессе эксплуатации и обслуживания экземпляров уведомления могут быть использованы для информирования о исключениях или состояниях, требующих особого внимания. Когда экземпляр сталкивается с аномалией или достигает состояния предупреждения, уведомления могут быть автоматически вызваны, чтобы помочь вам быстро выявить и локализовать проблемы.

Для повышения эффективности эксплуатации и обслуживания системы платформа разработала правила уведомлений на основе индикаторов мониторинга, используемых при устранении проблем с общими экземплярами, классифицируя и обобщая их в встроенные стратегии уведомлений.

Платформа поддерживает правила уведомлений на основе предопределенных индикаторов мониторинга и индикаторов мониторинга, определяемых пользователем. Когда происходят аномалии ресурсов или достигается состояние предупреждения, уведомления могут быть автоматически вызваны. В сочетании с функциональностью уведомлений платформы информация об уведомлениях может активно передаваться персоналу по эксплуатации и обслуживанию, напоминая им вовремя решить проблемы с уведомляемыми ресурсами, чтобы обеспечить бесперебойную работу бизнеса на платформе.

Для упрощения настройки уведомлений для большого количества ресурсов на платформе, платформа поддерживает настройку стандартизированных конфигураций уведомлений для аналогичных экземпляров через шаблоны уведомлений, позволяя вам быстро создавать стратегии уведомлений для ресурсов на основе существующих шаблонов.

## Типы уведомлений

- **Уведомление по метрикам**: Общие индикаторы мониторинга, извлекаемые платформой, которые удовлетворяют большинство потребностей клиентов. Вы можете настраивать уведомления, выбирая индикаторы мониторинга и устанавливая условия триггера. Когда данные мониторинга соответствуют условиям триггера правила уведомлений, будет вызвано уведомление.

- **Пользовательское уведомление**: Вы можете добавить правила метрик, специфичные для предприятия, на основе реальных случаев использования, чтобы лучше удовлетворить продвинутые потребности предприятия в уведомлениях на основе уведомлений по метрикам.

## Описание статуса

- Статус уведомления

  - **Уведомление**: Данные, полученные из запроса конфигурации правил уведомлений, соответствуют условиям триггера и вызвали уведомление.

  - **Ожидание**: Данные мониторинга, полученные из запроса конфигурации правил уведомлений, больше или равны *пороговому значению уведомления* в условиях триггера, но продолжительность еще не достигла условий триггера, что представляет собой критическое состояние перед вызовом уведомления. Например: Если условие триггера правила уведомления — это `использование CPU превышает 80% и продолжается 3 минуты`, когда система впервые определяет, что использование CPU превышает 80%, она отметит правило как `Ожидание`. Оно будет продолжать оцениваться, и если использование CPU останется выше 80% в течение 3 минут, статус правила изменится на **Уведомление**; если позже при оценке использование CPU упадет ниже 80%, статус правила вернется к **Нормально**.

  - **Нормально**: Данные, полученные из запроса конфигурации правил уведомлений, не соответствуют пороговому значению уведомления.

- Тихий статус

  - **Тихое ожидание**: Статус перед началом тишины после ее установки. В этом состоянии, когда правила в стратегии вызывают уведомление, уведомления будут отправляться нормально.

  - **Тишина**: Статус с начала тишины до конца тишины. В этом состоянии, когда правила в стратегии вызывают уведомление, уведомления не будут отправляться.

## Уведомления в реальном времени

Платформа отображает количество ресурсов, находящихся под уведомлением, и подробную информацию об уведомлениях через уведомления в реальном времени, позволяя персоналу по эксплуатации и обслуживанию и администраторам понимать общее состояние уведомлений бизнеса на платформе в реальном времени, быстро выявляя и устраняя неисправности, чтобы обеспечить нормальную работу платформы.

## История уведомлений

Для помощи персоналу по эксплуатации и обслуживанию и администраторам в анализе недавних условий мониторинга уведомлений и отслеживания исторических проблем, платформа поддерживает просмотр исторических записей уведомлений за определенный период времени. Доступная для просмотра историческая информация включает время появления, правила уведомлений, неисправные ресурсы, методы уведомления, статус и получателей уведомлений и т. д.

**Примечание**: Время хранения истории уведомлений совпадает с временем хранения событий и может быть обновлено в разделе **Управление платформой** путем изменения параметров конфигурации компонента хранения журналов.

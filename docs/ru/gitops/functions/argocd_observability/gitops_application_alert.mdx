# GitOps Applications Ops

## Обзор

Возможности управления **GitOps Applications** веб-консоли позволяют просматривать мониторинг **GitOps Applications**, логи и события. Также вы можете создавать политики оповещения для **GitOps Applications**. При возникновении аномалий в **GitOps Applications** будут инициированы проактивные уведомления, чтобы ускорить идентификацию, анализ и решение проблем.

## Предварительные условия

- **GitOps** приложение было создано в веб-консоли. [Создание Argo CD приложения через веб-консоль](../../how_to/create_gitops_app.mdx)
- [Установка плагинов мониторинга](/observability/monitor/install_monitor.mdx)

## Уведомление

> Создайте правило уведомления заранее, чтобы настроить правила. Когда **GitOps** приложения сталкиваются с аномалиями, будут инициированы проактивные уведомления для быстрой идентификации, анализа и решения проблем.

1. **Платформа контейнеров**, нажмите на **GitOps Applications**.

2. Выберите имя **GitOps** приложения из списка, для которого вы хотите создать правило уведомления.

3. Перейдите на вкладку **Уведомления**.

4. Нажмите на **Создать правило** и заполните основную информацию по требованиям.

5. Нажмите на **Добавить условие уведомления**, перейдите на страницу **Условия уведомлений**. Соответствующие описания метрик следующие:

:::info
Для других параметров конфигурации и настроек уведомлений обращайтесь к [Управлению Уведомлениями](/observability/monitor/functions/manage_alert.mdx).
:::

| **Название метрики**                                              | **Описание правила**                                                                                                       |
| :--------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------- |
| Состояние здоровья **GitOps** приложения `gitops.applicationset.healthy` | Состояние здоровья **GitOps** приложения:<br /> - 0: Неизвестно, Потеряно, Ухудшено или Приостановлено <br /> - 1: Синхронизация <br /> - 2: Здоровое |
| Состояние синхронизации **GitOps** приложения `gitops.applicationset.synced`    | Состояние синхронизации **GitOps** приложения:<br /> - 0: Синхронизация не удалась или Ожидание<br /> - 1: Синхронизация <br /> - 2: Синхронизировано                 |

6. Нажмите **Создать**.

## Логи

> Просмотрите логи для всех рабочих ресурсов, созданных **GitOps** приложениями. Логи позволяют быстро идентифицировать информацию о сбоях системы без зависимостей от плагина кластера **Log**.

- На странице деталей приложения **GitOps**, в разделе **Ресурсы Kubernetes**, нажмите на любое имя рабочей нагрузки, чтобы увидеть информацию **Логи** для этого ресурса справа.

## События

> Просмотрите события для всех ресурсов, распределенных **GitOps Applications**. События позволяют быстро идентифицировать информацию о сбоях системы без зависимостей от плагина кластера **Log**.

- На странице деталей приложения **GitOps**, перейдите на вкладку **События**, чтобы просмотреть агрегированные события для всех ресурсов.
- На странице деталей приложения **GitOps**, в разделе **Ресурсы Kubernetes**, нажмите на любое имя ресурса, чтобы увидеть события для этого ресурса справа.

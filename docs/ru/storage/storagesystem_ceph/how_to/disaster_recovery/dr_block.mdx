# Восстановление данных после катастроф в блочном хранилище

RBD Mirror — это функция блочного хранилища Ceph (RBD), которая позволяет выполнять асинхронную репликацию данных между различными кластерами Ceph, обеспечивая восстановление после катастроф (DR) между кластерами. Ее основная функция заключается в синхронизации данных в режиме основной-резервной копии, что гарантирует быструю передачу обслуживания обратно в резервный кластер в случае сбоя основного кластера.

:::warning

- RBD Mirror выполняет инкрементную синхронизацию на основе снимков с интервалом по умолчанию один раз в час (конфигурируемый). Дифференциальные данные между основными и резервными кластерами обычно соответствуют записям в пределах одного цикла создания снимков.
- RBD Mirror обеспечивает только резервное копирование данных хранилища и не обрабатывает резервные копии ресурсов Kubernetes. Пожалуйста, используйте функцию **Резервное копирование и восстановление** платформы для резервного копирования ресурсов PVC и PV.
  :::

## Терминология

| Термин                | Объяснение                                      |
| :-------------------- | :---------------------------------------------- |
| **Основной кластер**  | Кластер, который в данный момент предоставляет услуги хранения. |
| **Резервный кластер** | Резервный кластер, используемый для резервного копирования. |

## Конфигурация резервного копирования

### Предварительные условия

- Подготовьте два кластера, способных развернуть хранилище Alauda Container Platform (ACP) с Ceph: основной кластер и резервный кластер с сетевым соединением между ними.
- Оба кластера должны работать на одной и той же версии платформы (v3.12 или позднее).
- [Создайте распределенные сервисы хранения](../../installation/create_service_stand.mdx) в обоих кластерах — основном и резервном.
- Создайте пулы блочного хранения с **идентичными именами** в обоих кластерах — основном и резервном.
- Пожалуйста, убедитесь, что следующие три образа загружены в частный репозиторий образов платформы:
  - `quay.io/csiaddons/k8s-controller:v0.5.0`
  - `quay.io/csiaddons/k8s-sidecar:v0.8.0`
  - `quay.io/brancz/kube-rbac-proxy:v0.8.0`

### Процедуры

<Steps>
  ### Включите зеркалирование для пула блочного хранения основного кластера

  Выполните следующую команду на управляющем узле основного кластера:

  <Tabs>
    <Tab label="Команда">
      ```bash
      kubectl -n rook-ceph patch cephblockpool <block-pool-name> \
      --type merge -p '{"spec":{"mirroring":{"enabled":true,"mode":"image"}}}'
      ```
    </Tab>

    <Tab label="Вывод">
      ```bash
      cephblockpool.ceph.rook.io/<block-pool-name> patched
      ```
    </Tab>
  </Tabs>

  **Параметры**:

  - `<block-pool-name>`: Имя пула блочного хранения.

  ### <a id="blocktoken" />Получить токен пиринга

  Этот токен служит критической учетной записью для установления зеркальных соединений между кластерами.

  Выполните следующую команду на управляющем узле основного кластера:

  <Tabs>
    <Tab label="Команда">
      ```bash
      kubectl get secret -n rook-ceph \
      $(kubectl get cephblockpool.ceph.rook.io <block-pool-name> -n rook-ceph -o jsonpath='{.status.info.rbdMirrorBootstrapPeerSecretName}') \
      -o jsonpath='{.data.token}' | base64 -d
      ```
    </Tab>

    <Tab label="Вывод">
      ```bash
      # Вывод сокращен из-за конфиденциальной информации
      eyJmc2lkIjoiMjc2N2I3ZmEtY2YwYi00N...
      ```
    </Tab>
  </Tabs>

  **Параметры**:

  - `<block-pool-name>`: Имя пула блочного хранения.

  ### Создать секрет токена пиринга в резервном кластере

  Выполните следующую команду на управляющем узле резервного кластера:

  <Tabs>
    <Tab label="Команда">
      ```bash
      kubectl -n rook-ceph create secret generic rbd-primary-site-secret \
      --from-literal=token=<token> \
      --from-literal=pool=<block-pool-name>
      ```
    </Tab>

    <Tab label="Вывод">
      ```
      secret/rbd-primary-site-secret created
      ```
    </Tab>
  </Tabs>

  **Параметры**:

  - `<token>`: Токен, полученный из [Шага 2](#blocktoken).
  - `<block-pool-name>`: Имя пула блочного хранения.

  ### Включите зеркалирование для пула блочного хранения резервного кластера

  Выполните следующую команду на управляющем узле резервного кластера:

  <Tabs>
    <Tab label="Команда">
      ```bash
      kubectl -n rook-ceph patch cephblockpool <block-pool-name> --type merge -p \
      '{
        "spec": {
          "mirroring": {
            "enabled": true, 
            "mode": "image", 
            "peers": {
              "secretNames": [
                "rbd-primary-site-secret"
              ]
            }
          }
        }
      }'
      ```
    </Tab>

    <Tab label="Вывод">
      ```
      cephblockpool.ceph.rook.io/<block-pool-name> patched
      ```
    </Tab>
  </Tabs>

  **Параметры**:

  - `<block-pool-name>`: Имя пула блочного хранения.

  ### Развернуть демон зеркалирования в резервном кластере

  Этот демон отвечает за мониторинг и управление процессами синхронизации RBD Mirror, включая синхронизацию данных и обработку ошибок.

  Выполните следующую команду на управляющем узле резервного кластера:

  <Tabs>
    <Tab label="Команда">
      ```yaml
      cat << EOF | kubectl apply -f -
      apiVersion: ceph.rook.io/v1
      kind: CephRBDMirror
      metadata:
        name: rbd-mirror
        namespace: rook-ceph
      spec:
        count: 1
      EOF
      ```
    </Tab>

    <Tab label="Вывод">
      ```
      cephrbdmirror.ceph.rook.io/rbd-mirror created
      ```
    </Tab>
  </Tabs>

  ### Проверьте статус зеркалирования

  Выполните следующую команду на управляющем узле резервного кластера:

  <Tabs>
    <Tab label="Команда">
      ```bash
      kubectl get cephblockpools.ceph.rook.io <block-pool-name> -n rook-ceph -o jsonpath='{.status.mirroringStatus.summary}'
      ```
    </Tab>

    <Tab label="Вывод">
      ```bash
      # Все статусы "OK" указывают на нормальную работу
      {"daemon_health":"OK","health":"OK","image_health":"OK","states":{}}
      ```
    </Tab>
  </Tabs>

  **Параметры**:

  - `<block-pool-name>`: Имя пула блочного хранения.

  ### Включите репликацию бокового демона

  2. Включите боковой демон csi

  Выполните следующие команды на управляющих узлах основного и резервного кластеров:

  ```bash
  kubectl patch cm rook-ceph-operator-config -n rook-ceph --type json --patch \
  '[
    {
      "op": "add", 
      "path": "/data/CSI_ENABLE_OMAP_GENERATOR",

    {
      "op": "add",
  ```

  ### Создать VolumeReplicationClass

  Выполните следующие команды на управляющих узлах основного и резервного кластеров:

  <Tabs>
    <Tab label="Команда">
      ```yaml
      cat << EOF | kubectl apply -f -
      apiVersion: replication.storage.openshift.io/v1alpha1
      kind: VolumeReplicationClass
      metadata:
        name: rbd-volumereplicationclass
      spec:
        provisioner: rook-ceph.rbd.csi.ceph.com
        parameters:
          mirroringMode: snapshot
          schedulingInterval: "<scheduling-interval>" # [!code callout]
          replication.storage.openshift.io/replication-secret-name: rook-csi-rbd-provisioner
          replication.storage.openshift.io/replication-secret-namespace: rook-ceph
      EOF
      ```
    </Tab>

    <Tab label="Вывод">
      ```bash
      volumereplicationclass.replication.storage.openshift.io/rbd-volumereplicationclass created
      ```
    </Tab>
  </Tabs>

  <Callouts>
    1. `<scheduling-interval>`: Интервал планирования (например, schedulingInterval: "1h" указывает на выполнение каждую 1 час).
  </Callouts>

  ### Включите зеркалирование для PVC

  Выполните следующую команду на управляющем узле основного кластера:

  <Tabs>
    <Tab label="Команда">
      ```yaml
      cat << EOF | kubectl apply -f -
      apiVersion: replication.storage.openshift.io/v1alpha1
      kind: VolumeReplication
      metadata:
        name: <vr-name> # [!code callout]
        namespace: <namespace> # [!code callout]
      spec:
        autoResync: false
        volumeReplicationClass: rbd-volumereplicationclass
        replicationState: primary
        dataSource:
          apiGroup: ""
          kind: PersistentVolumeClaim
          name: <pvc-name> # [!code callout]
      EOF
      ```
    </Tab>

    <Tab label="Вывод">
      ```bash
      volumereplication.replication.storage.openshift.io/<mirror-pvc-name> created
      ```
    </Tab>
  </Tabs>

  <Callouts>
    1. `<vr-name>`: Имя объекта VolumeReplication, рекомендуется, чтобы оно совпадало с именем PVC.
    2. `<namespace>`: Пространство имен, к которому принадлежит VolumeReplication, должно совпадать с пространством имен PVC.
    3. `<pvc-name>`: Имя PVC, для которого необходимо включить зеркалирование.
  </Callouts>

  **Примечание**
  После включения изображение RBD в резервном кластере становится доступным только для чтения.
</Steps>

## Переход на резервный кластер

Когда основной кластер выходит из строя, необходимо переключить отношения основной и резервной копии изображения RBD.

### Предварительные условия

- Ресурсы Kubernetes основного кластера были скопированы и восстановлены в резервный кластер, включая PVC, PV, рабочие нагрузки приложений и т. д.

### Процедуры

<Steps>
  #### Создать VolumeReplication

  Выполните следующую команду на управляющем узле резервного кластера:

  ```yaml
  cat << EOF | kubectl apply -f -
  apiVersion: replication.storage.openshift.io/v1alpha1
  kind: VolumeReplication
  metadata:
    name: <vr-name> # [!code callout]
    namespace: <namespace> # [!code callout]
  spec:
    autoResync: false
    dataSource:
      apiGroup: ""
      kind: PersistentVolumeClaim
      name: <mirror-pvc-name> # [!code callout]
    replicationHandle: ""
    replicationState: primary
    volumeReplicationClass: rbd-volumereplicationclass
  EOF
  ```

  <Callouts>
    1. `<vr-name>`: Имя VolumeReplication.
    2. `<namespace>`: Пространство имен PVC.
    3. `<mirror-pvc-name>`: Имя PVC.
  </Callouts>

  **Примечание**
  После создания изображение RBD в резервном кластере становится основным и доступным для записи.
</Steps>

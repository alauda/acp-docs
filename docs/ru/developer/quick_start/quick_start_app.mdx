# Создание простого приложения через образ

> Этот технический гид демонстрирует, как эффективно создавать, управлять и получать доступ к контейнеризованным приложениям в <Term name="product" textCase="capitalize" /> с использованием методов, основанных на Kubernetes.

## Введение

### Сценарии использования

- Новые пользователи, стремящиеся понять основные рабочие процессы создания приложений на платформах Kubernetes
- Практическое упражнение, демонстрирующее основные возможности платформы, включая:
  - Оркестрация проектов/неймспейсов
  - Создание развертываний
  - Шаблоны экспозиции сервисов
  - Верификация доступности приложения

### Временные затраты

Оценочное время выполнения: 10-15 минут

## Процедура

### Создать неймспейс \{#namespace}

> Неймспейсы обеспечивают логическую изоляцию для группировки ресурсов и управления квотами.

**Предварительные требования**

- Разрешения на создание, обновление и удаление неймспейсов (например, роли Администратора или Администратора проекта)
- Настроенный kubectl с доступом к кластеру

**Процесс создания**

1. Войдите в систему и перейдите в **Управление проектом** > **Неймспейсы**

2. Выберите **Создать неймспейс**

3. Настройте основные параметры:

   | \*\* Параметр \*\* | **Описание**                                     |
   | :------------------ | :------------------------------------------------- |
   | Кластер             | Целевой кластер из связанных с проектом кластеров |
   | Неймспейс           | Уникальный идентификатор (авто-префикс с именем проекта) |

4. Завершите создание с помощью стандартных ограничений ресурсов

### Настроить репозиторий изображений \{#image}

<Term name="product" textCase="capitalize" /> поддерживает несколько стратегий источников изображений:

#### Метод 1: Интегрированный реестр через инструментальную цепочку

1. Перейдите в **Управление платформой** > **Инструментальная цепочка** > **Интеграция**

2. Начните новую интеграцию:

   | Параметр    | Требование                                  |
   | ------------ | ---------------------------------------- |
   | Имя         | Уникальный идентификатор интеграции      |
   | API-адрес   | URL реестра (HTTP/HTTPS)                  |
   | Секрет      | Существующая или новосозданная учетная запись |

3. Назначьте реестр целевому проекту платформы

#### Метод 2: Внешние услуги реестра

- Используйте публично доступные URL реестров (например, Docker Hub)
- Пример: `index.docker.io/library/nginx:latest`

**Требование к проверке**

- Сетевая инфраструктура кластера должна иметь доступ к egress к конечным точкам реестра

### Создать приложение через развертывание \{#deployment}

> Развертывания обеспечивают декларативные обновления для под-реплик.

**Процесс создания**

1. На экране **Платформы контейнеров**:
   - Используйте селектор неймспейса, чтобы выбрать целевую границу изоляции
2. Перейдите в **Рабочие нагрузки** > **Развертывания**
3. Нажмите **Создать развертывание**
4. Укажите источник изображения:
   - Выберите интегрированный реестр *или*
   - Введите URL внешнего изображения (например, `index.docker.io/library/nginx:latest`)
5. Настройте идентичность рабочей нагрузки и запустите

**Операции управления**

- Мониторинг состояния реплик
- Просмотр событий и логов
- Проверка YAML-манифестов
- Анализ метрик ресурсов, оповещений

### Опубликовать сервис через NodePort \{#service}

Сервисы обеспечивают сетевую доступность групп подов.

**Процесс создания**

1. Перейдите в **Сетевые подключения** > **Сервисы**

2. Нажмите **Создать сервис** с параметрами:

   | Параметр    | Значение                                      |
   | :----------- | :------------------------------------------- |
   | Тип         | NodePort                                     |
   | Селектор     | Имя целевого развертывания               |
   | Маршрутизация портов | Порт сервиса: Порт контейнера (например, 8080:80) |

3. Подтвердите создание.

**Критически**

- Виртуальный IP, видимый в кластере
- Диапазон выделения NodePort (30000-32767)

Внутренние маршруты позволяют рабочим нагрузкам обнаруживать сервисы, предоставляя объединенный IP-адрес или порт хоста для доступа.

1. Нажмите на **Сеть** > **Сервис**.

2. Нажмите на **Создать сервис**.

3. Настройте **Детали** на основе нижеприведенных параметров, оставив другие параметры по умолчанию.

   | Параметр     | Описание                                                                                                                                                                                                                |
   | ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
   | Имя          | Введите имя сервиса.                                                                                                                                                                                             |
   | Тип          | `NodePort`                                                                                                                                                                                                                 |
   | Имя рабочей нагрузки | Выберите `Развертывание`, созданное ранее.                                                                                                                                                                                |
   | Порт          | **Порт сервиса**: номер порта, открываемого сервисом в кластере, то есть Порт, например, `8080`.<br />**Порт контейнера**: номер порта (или имя), сопоставленный с портом сервиса, то есть targetPort, например, `80`. |

4. Нажмите на **Создать**. На этом этапе сервис успешно создан.

### Проверить доступность приложения \{#application}

**Метод проверки**

1. Получите компоненты открытой конечной точки:
   - **IP узла**: публичный адрес рабочего узла
   - **NodePort**: выделенный внешний порт
2. Составьте URL для доступа: `http://<Node_IP>:<NodePort>`
3. Ожидаемый результат: страница приветствия Nginx

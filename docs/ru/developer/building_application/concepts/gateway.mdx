# Шлюз

Входной шлюз (Gateway) — это экземпляр, развернутый из класса шлюза. Он создает слушателей для захвата внешнего трафика на указанных доменных именах и портах. Вместе с правилами маршрутизации он может направлять указанный внешний трафик к соответствующим бэкенд-экземплярам.

## Терминология

| Название ресурса    | Обзор                                                                                                                                                                                                                                                                                                                                                          | Инструкции по использованию                                                       |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| **Класс шлюза**     | В стандартной документации API шлюза класс шлюза определяется как шаблон для создания шлюзов. Разные шаблоны могут создавать входные шлюзы для различных бизнес-сценариев, что облегчает быстрое управление трафиком.                                                                                                                                               | Платформа включает в себя специализированные классы шлюзов.                      |
| **Входной шлюз**    | Входной шлюз соответствует конкретным экземплярам ресурсов, и пользователи могут эксклюзивно использовать все ресурсы прослушивания и вычислений этого входного шлюза. Это конфигурация правил маршрутизации, действительная для слушателя. Когда внешний трафик обнаруживается шлюзом, он распределяется к бэкенд-экземплярам в соответствии с правилами маршрутизации. | Его можно рассматривать как экземпляр балансировщика нагрузки.                   |
| **Правило маршрута**| Правила маршрутизации определяют ряд рекомендаций для распределения трафика от шлюза к сервисам. В настоящее время стандартные поддерживаемые типы правил маршрутизации в API шлюза включают HTTPRoute, TCPRoute, UDPRoute и т. д.                                                                                                                                           | Платформа в настоящее время поддерживает прослушивание протоколов HTTP, HTTPS, TCP и UDP. |

## Функциональный фон

Объекты ресурсов Ingress не удовлетворяют сетевым требованиям, что приводит к множеству сценариев, в которых контроллерам Ingress требуется определение аннотаций или CRD для функционального расширения. Это крайне негативно сказывается на стандартизации и миграции данных.

Проект с открытым исходным кодом API шлюза, управляемый сообществом SIG-NETWORK (ранее известным как Service API), направлен на улучшение сетевого взаимодействия сервисов через масштабируемый интерфейс на основе ролей. API шлюза представляет собой набор ресурсов API в Kubernetes, включая GatewayClass, Gateway, HTTPRoute, TCPRoute, Service и т. д. Эти ресурсы совместно создают бизнес-модели для различных сетевых сценариев.

## Процесс использования

Чтобы использовать функциональность входного шлюза, его необходимо использовать в сочетании со слушателями и правилами маршрутизации. Ниже приведен общий пример процесса в типичных сценариях:

1. Выберите класс шлюза.

2. Создайте входной шлюз и настройте слушателей для мониторинга трафика протоколов HTTP, HTTPS, TCP и UDP с указанных доменных имен.

3. Добавьте правила маршрутизации, чтобы трафик, обнаруженный шлюзом, перенаправлялся к указанным бэкенд-экземплярам в соответствии с правилами маршрутизации.

# Сравнение различных методов входящего трафика

Платформа ACP поддерживает несколько спецификаций входящего трафика в экосистеме Kubernetes. Этот документ сравнивает их (Service, Ingress, Gateway API и ALB Rule), чтобы помочь пользователям сделать правильный выбор.

## Для L4 (TCP/UDP) трафика

Сервисы типа LoadBalancer, Gateway API и ALB Rules могут все внешне открывать L4 трафик. Здесь мы рекомендуем использовать подход с сервисом типа LoadBalancer. Как Gateway API, так и ALB Rules реализованы с помощью ALB, который является прокси в пространстве пользователей, и их производительность значительно ухудшается при обработке L4 трафика по сравнению с сервисами типа LoadBalancer.

## Для L7 (HTTP/HTTPS) трафика

Хотя Ingress, GatewayAPI и ALB Rules могут все внешне открывать L7 трафик, они отличаются своими возможностями и моделями изоляции.

### Ingress

Ingress является стандартной спецификацией, принятой сообществом Kubernetes, и рекомендуется для использования по умолчанию. Ingress обрабатывается экземплярами ALB, которые управляются администратором платформы.

### GatewayAPI

GatewayAPI предоставляет более гибкий режим изоляции, однако он не так зрел, как Ingress. Используя GatewayAPI, разработчик может создать свои собственные изолированные экземпляры ALB для обработки правил GatewayAPI. Таким образом, если вам нужно делегировать создание и управление экземплярами ALB разработчикам, необходимо выбрать использование GatewayAPI.

### ALB Rule

ALB Rule (Load Balancer в пользовательском интерфейсе) предоставляет наиболее гибкие правила сопоставления трафика и наибольшие возможности. На самом деле, как Ingress, так и GatewayAPI реализуются путем преобразования их в ALB Rules. Однако ALB Rule более сложен, чем Ingress и GatewayAPI, и не является стандартным API сообщества. Поэтому мы рекомендуем использовать его только в том случае, если Ingress и GatewayAPI не соответствуют вашим потребностям.

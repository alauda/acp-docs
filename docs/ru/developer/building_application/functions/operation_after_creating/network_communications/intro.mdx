# Сравнение различных методов Ingress

Платформа ACP поддерживает несколько спецификаций входящего трафика в экосистеме Kubernetes. Этот документ сравнивает их (Service, Ingress, Gateway API и ALB Rule), чтобы помочь пользователям сделать правильный выбор.

## Для трафика L4 (TCP/UDP)

Сервисы типа LoadBalancer, Gateway API и ALB Rules могут обнаруживать трафик L4 вовне. Здесь мы рекомендуем использовать подход с сервисом типа LoadBalancer. 
И Gateway API, и ALB Rules реализованы с помощью ALB, который является прокси-сервером пользовательского пространства, и их производительность значительно ухудшается при обработке трафика L4 по сравнению с сервисами типа LoadBalancer.

## Для трафика L7 (HTTP/HTTPS)

Хотя Ingress, GatewayAPI и ALB Rules могут все обнаруживать трафик L7 вовне, они различаются по своим возможностям и моделям изоляции.

### Ingress

Ingress является стандартной спецификацией, принятой сообществом Kubernetes, и рекомендуется для использования по умолчанию. 
Ingress обрабатывается экземплярами ALB, которые управляются администратором платформы.

### GatewayAPI

GatewayAPI предоставляет более гибкий режим изоляции, однако он не так развит, как Ingress. 
Используя GatewayAPI, разработчики могут создавать свои собственные изолированные экземпляры ALB для обработки правил GatewayAPI. 
Поэтому, если вам нужно делегировать создание и управление экземплярами ALB разработчикам, вам необходимо выбрать использование GatewayAPI.

### ALB Rule

ALB Rule (Load Balancer в интерфейсе) предоставляет самые гибкие правила сопоставления трафика и наибольшие возможности. На самом деле, как Ingress, так и GatewayAPI реализованы путем их преобразования в ALB Rules. 
Тем не менее, ALB Rule более сложен, чем Ingress и GatewayAPI, и не является стандартным API сообщества. Поэтому мы рекомендуем использовать его только тогда, когда Ingress и GatewayAPI не удовлетворяют вашим потребностям.

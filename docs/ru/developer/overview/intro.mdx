# Введение

Модуль представления разработчика предоставляет разработчикам возможности оркестрации облачных приложений и управления операциями. Он обеспечивает единый интерфейс для композиции приложений из нескольких источников, интегрируя встроенные инструменты наблюдаемости для операций в производственной среде.

## Преимущества

> Модуль представления разработчика предлагает следующие ключевые преимущества:

1. Единая оркестрация приложений

   - Образы: Развертывание из публичных/частных реестров с изображением

   - YAML: Прямые декларации ресурсов Kubernetes с верификацией схемы

   - Source to Image (S2I): Создание контейнеризованных приложений непосредственно из исходного кода

   - Helm Charts: Развертывание упакованных приложений из кураторского каталога приложений

   - Реализация композиции приложений, согласованной с GitOps, с использованием нескольких подходов

2. Холистическое управление жизненным циклом

> Реализация декларативного управления рабочими нагрузками и пространствами имен:

- Прогрессивная доставка: Deployment с канарейками/Blue-Green через ServiceMesh

- Управление ресурсами:

  - Обеспечение пространства имен с политиками RBAC

  - Политики распределения ресурсов через HPA/VPA

  - Динамическое масштабирование с интеграцией Cluster Autoscaler

- Автоматизация рабочих процессов: Интеграция CI/CD пайплайна с Tekton

3. Управление пространством имен предприятия

> Реализация управления многоарендными пространствами имен:

- Полное управление жизненным циклом

- Гарантии ресурсов:

  - Конфигурации ResourceQuota и LimitRange

  - Настраиваемые коэффициенты превышения для CPU/Memory

4. Полная наблюдаемость стека

> Интегрированный стек мониторинга с:

- Корреляция событий: Интеграция событий Kubernetes и журналов аудита
- Аналитика журналов: Сбор журналов
- Панель мониторинга метрик: Мониторинг и настроенные правила оповещения

## Случаи использования

> Основные случаи использования модуля разработчика включают в себя:

- Многооблачное развертывание

Организации распределяют рабочие нагрузки между несколькими облачными провайдерами (AWS, Azure, GCP), чтобы избежать привязки к поставщику, оптимизировать затраты и обеспечить устойчивость. Доставка облачных приложений обеспечивает согласованные пайплайны развертывания, которые абстрагируют реализации, специфичные для провайдера.

- Гибридные облачные среды

Предприятия поддерживают локальную инфраструктуру наряду с ресурсами публичного облака. Доставка облачных приложений обеспечивает единое развертывание приложений в гибридных средах, управляя сложностями неоднородной инфраструктуры.

- Интеграция с краевыми вычислениями

По мере того как краевые вычисления набирают популярность, приложения должны работать в централизованных облаках, краевых устройствах и региональных узлах. Доставка облачных приложений расширяет возможности развертывания для этих распределенных краевых сред.

- Пайплайн разработки в производство

Облачные методологии обеспечивают бесшовное продвижение приложений от разработки через тестирование/стадию до производства, сохраняя согласованность конфигурации, учитывая специфические требования среды.

- Глобальные многорегиональные развертывания

Для глобально распределенных приложений доставка облачных приложений обеспечивает согласованные развертывания в различных географических регионах, отвечая требованиям оптимизации задержки и соблюдения локальности данных.

- Восстановление после катастроф и непрерывность рабочих нагрузок

Доставка облачных приложений облегчает создание среды восстановления после катастроф, которая дублирует производственные системы, позволяя быстро переключаться и обеспечивая непрерывную работу.

## Основополагающие облачно-нативные принципы

> Эти сценарии используют ключевые облачно-нативные принципы:

- Контейнеризация
- Инфраструктура как код (IaC)
- Декларативные конфигурации
- Невозможность изменения инфраструктуры
- Рабочие процессы GitOps

Эти принципы обеспечивают консистентность, надежность и автоматизацию во множестве различных вычислительных сред.

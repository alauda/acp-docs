# Введение

Модуль Developer view предоставляет разработчикам возможности оркестрации и управления облачными приложениями. Он предлагает единый интерфейс для компоновки приложений из различных источников, одновременно интегрируя встроенные инструменты наблюдаемости для операций в производственной среде.

## Преимущества

> Модуль Developer view предоставляет следующие ключевые преимущества:

1. Единая оркестрация приложений

   - Образы: Развертывание из публичных/частных реестров с изображением

   - YAML: Прямые декларации ресурсов Kubernetes с валидацией схемы

   - Source to Image (S2I): Создание контейнеризированных приложений непосредственно из исходного кода

   - Helm Charts: Развертывание упакованных приложений из кураторского каталога приложений

   - Реализует компоновку приложений в соответствии с GitOps, используя несколько подходов

2. Комплексное управление жизненным циклом

> Реализует декларативное управление рабочими нагрузками и пространствами имен:

- Пошаговая доставка: Развертывания Canary/Blue-Green через ServiceMesh

- Управление ресурсами:

  - Предоставление пространства имен с политиками RBAC

  - Политики распределения ресурсов через HPA/VPA

  - Динамическое масштабирование с интеграцией Cluster Autoscaler

- Автоматизация рабочих процессов: Интеграция CI/CD конвейеров с Tekton

3. Контроль пространств имен уровня предприятия

> Реализует управление пространствами имен для нескольких арендаторов:

- Полное управление жизненным циклом

- Гарантии ресурсов:

  - Конфигурации ResourceQuota и LimitRange

  - Конфигурируемые коэффициенты перерасхода для CPU/Памяти

4. Полная наблюдаемость

> Интегрированная стек наблюдения с:

- Корреляция событий: интеграция событий Kubernetes и журнала аудита
- Аналитика журналов: агрегация журналов
- Дашборд метрик: Мониторинг и пользовательские правила оповещения

## Сценарии применения

> Основные сценарии использования модуля Developer включают:

- Мультиоблачное развертывание

Организации распределяют рабочие нагрузки между несколькими облачными провайдерами (AWS, Azure, GCP), чтобы избежать привязки к поставщику, оптимизировать затраты и обеспечить устойчивость. Доставка облачных приложений позволяет создать консистентные конвейеры развертывания, абстрагирующие реализации, специфичные для поставщика.

- Гибридные облачные среды

Предприятия поддерживают локальную инфраструктуру наряду с ресурсами публичного облака. Доставка облачных приложений обеспечивает единое развертывание приложений в гибридных средах, управляя сложностями неоднородной инфраструктуры.

- Интеграция с периферийными вычислениями

Поскольку периферийные вычисления становятся все более важными, приложения должны работать как в централизованных облаках, так и на периферийных устройствах и региональных узлах. Доставка облачных приложений расширяет возможности развертывания для этих распределенных периферийных сред.

- Конвейер разработки в производство

Методологии облачной нативности обеспечивают плавную передачу приложений от разработки через тестирование/этапирование к производству, сохраняя согласованность конфигурации и учитывая специфические требования среды.

- Глобальные многорегиональные развертывания

Для глобально распределенных приложений доставка облачных приложений обеспечивает консистентные развертывания в различных географических регионах, учитывая оптимизацию задержек и соответствие требованиям локализации данных.

- Восстановление после сбоя и непрерывность рабочей нагрузки

Доставка облачных приложений облегчает предоставление среды для восстановления после сбоя, которая отражает производственные системы, обеспечивая быструю переадаптацию и непрерывную работу.

## Кросс-функциональные принципы облачной нативности

> Эти сценарии используют ключевые принципы облачной нативности:

- Контейнеризация
- Инфраструктура как код (IaC)
- Декларативные конфигурации
- Невозможная инфраструктура
- GitOps рабочие процессы

Эти принципы обеспечивают согласованность, надежность и автоматизацию в различных вычислительных средах.

# Аутентификация (Auth)

## Основное понятие

### Что такое аутентификация

Аутентификация (Auth) – это механизм, выполняющий проверку подлинности до того, как запрос достигнет фактического сервиса. Это позволяет вам обрабатывать аутентификацию на уровне ALB единообразно, не внедряя логику аутентификации в каждый бэкенд-сервис.

### Поддерживаемые методы аутентификации

ALB поддерживает два основных метода аутентификации:

1. **Forward Auth (Внешняя аутентификация)**
   - Отправляет запрос к внешнему сервису аутентификации для проверки подлинности пользователя
   - Применимые сценарии: необходимость в сложной логике аутентификации, такой как OAuth, SSO и т. д.
   - Рабочий процесс:
     1. Запрос пользователя поступает в ALB
     2. ALB пересылает информацию о аутентификации в сервис аутентификации
     3. Сервис аутентификации возвращает результат проверки
     4. В зависимости от результата аутентификации решается, разрешить ли доступ к бэкенд-сервису

2. **Basic Auth (Базовая аутентификация)**
   - Простая система аутентификации, основанная на имени пользователя и пароле
   - Применимые сценарии: простое управление доступом, защита среды разработки
   - Рабочий процесс:
     1. Запрос пользователя поступает в ALB
     2. ALB проверяет имя пользователя и пароль в запросе
     3. Сравнивает с конфигурированной информацией для аутентификации
     4. Если проверка проходит, пересылает запрос в бэкенд-сервис

### Методы настройки аутентификации

1. **Глобальная аутентификация**
   - Настраивается на уровне ALB, применяется ко всем сервисам
   - Настраивается на уровне ALB или FT CR

2. **Аутентификация на уровне пути**
   - Настраивается на конкретном пути Ingress
   - Настраивается на конкретном правиле
   - Может переопределять глобальную конфигурацию аутентификации

3. **Отключение аутентификации**
   - Отключает аутентификацию для конкретного пути
   - Настраивается на Ingress с аннотацией: `alb.ingress.cpaas.io/auth-enable: "false"`
   - Настраивается на Правиле с CR

### Обработка результатов аутентификации

- Успех аутентификации: запрос будет переслан в бэкенд-сервис
- Провал аутентификации: возвращает ошибку 401 неавторизованный
- Можно настроить поведение перенаправления после неудачи аутентификации (применимо к внешней аутентификации)

## Быстрый старт

Настройка базовой аутентификации с ALB

### Развертывание ALB

```bash
cat <<EOF | kubectl apply -f -
apiVersion: crd.alauda.io/v2
kind: ALB2
metadata:
  name: auth
  namespace: cpaas-system
spec:
  config:
    networkMode: container
    projects:
    - ALL_ALL
    replicas: 1
    vip: 
      enableLbSvc: false
  type: nginx
EOF
export ALB_IP=$(kubectl get pods -n cpaas-system -l service_name=alb2-auth -o jsonpath='{.items[*].status.podIP}');echo $ALB_IP
```

### Настройка секрета и Ingress


## forward-auth

```mermaid
sequenceDiagram
client ->> alb: запрос клиента (cli-request)
alb ->> auth-server: запрос ALB к сервису аутентификации (auth-request)
auth-server ->> alb: ответ сервиса аутентификации 200 (auth-response)
alb ->> app: запрос к приложению
app ->> alb: ответ приложения
alb ->> client: ответ клиента
```

Связанные аннотации:

- nginx.ingress.kubernetes.io/auth-url
- nginx.ingress.kubernetes.io/auth-method
- nginx.ingress.kubernetes.io/auth-signin
- nginx.ingress.kubernetes.io/auth-signin-redirect-param
- nginx.ingress.kubernetes.io/auth-response-headers
- nginx.ingress.kubernetes.io/auth-proxy-set-headers
- nginx.ingress.kubernetes.io/auth-request-redirect
- nginx.ingress.kubernetes.io/auth-always-set-cookie

Эти аннотации описывают изменения, внесенные в auth-request, app-request и cli-response на приведенной выше диаграмме.

### Конструирование связанных аннотаций

#### auth-url

URL для auth-request, значение может быть переменной.

#### auth-method

Метод для auth-request.

#### auth-proxy-set-headers

Значение – это ссылка на ConfigMap в формате `ns/name`.
По умолчанию все заголовки из cli-request будут отправлены на сервис аутентификации. Дополнительные заголовки могут быть настроены через proxy\_set\_header. Следующие заголовки отправляются по умолчанию:

```
X-Original-URI          $request_uri;
X-Scheme                $pass_access_scheme;
X-Original-URL          $scheme://$http_host$request_uri;
X-Original-Method       $request_method;
X-Sent-From             "alb";
X-Real-IP               $remote_addr;
X-Forwarded-For         $proxy_add_x_forwarded_for;
X-Auth-Request-Redirect $request_uri;	
```

### Конструирование аннотаций, связанных с app-request

#### auth-response-headers

Значение – это строка, разделенная запятыми, позволяющая перенести конкретные заголовки из auth-response в app-request.
пример:

```
nginx.ingress.kubernetes.io/auth-response-headers: Remote-User,Remote-Name
```

Когда ALB инициирует app-request, он будет включать Remote-User и Remote-Name из заголовков auth-response.

### Обработка файлов cookie

auth-response и app-response могут оба устанавливать файлы cookie. По умолчанию только когда app-response.success, значение auth-response.set-cookie будет объединено с cli-response.set-cookie.

```mermaid
flowchart TD
    subgraph "не включать always-set-cookie"
        A[app-response.set-cookie] --> C{app-response.success?}
        B[auth-response.set-cookie] --> C

    subgraph "включить always-set-cookie"

    end
```

### Перенаправление для настройки, связанной с подпиской

Когда сервис аутентификации возвращает 401, мы можем установить заголовок перенаправления в cli-response, чтобы указать браузеру перенаправить на URL, указанный в auth-signin для проверки.

```mermaid
sequenceDiagram
client ->> alb: запрос клиента (cli-request)
alb ->> auth-server: запрос ALB к сервису аутентификации (auth-request)
auth-server ->> alb: ответ сервиса аутентификации не 200 (auth-response)
alb ->> client: в случае неудачи аутентификации, alb отвечает cli-response (с заголовком location для перенаправления)
```

#### auth-signin

Значение – это URL, указывающее заголовок location в cli-response.

#### auth-signin-redirect-param

Имя параметра запроса в signin-url, по умолчанию rd.
если signin-url не содержит `auth-signin-redirect-param` указанный имя параметра, alb автоматически добавит параметр. Значение параметра будет установлено на `$pass_access_scheme://$http_host$escaped_request_uri`, используемое для записи оригинального URL запроса.

#### auth-request-redirect

Установить заголовок `x-auth-request-redirect` в auth-request.

## basic-auth

basic-auth является процессом аутентификации, описанным в [RFC 7617](https://datatracker.ietf.org/doc/html/rfc7617).
Взаимодействие проходит следующим образом:

```mermaid
sequenceDiagram
client ->> alb: запрос клиента (cli-request)
alb ->> alb : проверка имени пользователя и пароля в запросе
alb ->> client: cli-response
```

### auth-realm

[описание защищенной области](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Headers/WWW-Authenticate#realm)
Это значение области в заголовке `WWW-Authenticate` cli-response.
WWW-Authenticate: Basic realm="$realm"

### auth-type

Тип схемы аутентификации, в данный момент поддерживается только базовый

### auth-secret

Секретный ссылок имени пользователя и пароля, формат ns/name

### auth-secret-type

Секрет поддерживает два типа:

1. auth-file: данные секрета содержат только один ключ "auth", а его значение - строка формата Apache htpasswd. например:
   ```
   data:
     auth: "user1:$apr1$xyz..."
   ```

2. auth-map: данные секрета каждое ключ представляет собой имя пользователя, а соответствующее значение – хэш пароля (без имени пользователя в формате htpasswd). например:
   ```
   data:
     user1: "$apr1$xyz...."
     user2: "$apr1$abc...."
   ```

Примечание: В данный момент поддерживаются только хэши паролей формата htpasswd, сгенерированные с использованием алгоритма apr1.

## CR

ALB CR добавил элементы конфигурации, связанные с аутентификацией, которые могут быть настроены на ALB/Frontend/Rule CR.
Во время выполнения ALB конвертирует аннотации на Ingress в правила.

```yaml
auth:
  # Конфигурация базовой аутентификации
  basic:
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-type: basic
    auth_type: "basic"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-realm
    realm: "Ограниченный доступ"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-secret
    secret: "ns/name"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-secret-type
    secret_type: "auth-map|auth-file"
  # Конфигурация внешней аутентификации
  forward:
    #  boolean; соответствует nginx.ingress.kubernetes.io/auth-always-set-cookie
    always_set_cookie: true
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-proxy-set-headers
    auth_headers_cm_ref: "ns/name"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-request-redirect
    auth_request_redirect: "/login"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-method
    method: "GET"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-signin
    signin: "/signin"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-signin-redirect-param
    signin_redirect_param: "redirect_to"
    #  []string; соответствует nginx.ingress.kubernetes.io/auth-response-headers
    upstream_headers:
      - "X-User-ID" 
      - "X-User-Name"
      - "X-User-Email"
    #  строка; соответствует nginx.ingress.kubernetes.io/auth-url
    url: "http://auth-service/validate"
```

Аутентификация поддерживает настройку:

- Auth CR's `.spec.config.auth`
- Frontend CR's `.spec.config.auth`
- Rule CR's `.spec.config.auth`

Порядок наследования: Alb > Frontend > Rule. Если дочерний CR не настроен, будет использоваться конфигурация родительского CR.

## Специальные аннотации Ingress для ALB

В процессе обработки Ingress ALB определяет приоритет на основе префикса аннотации. Приоритет от высокого к низкому:

- `index.$rule_index-$path_index.alb.ingress.cpaas.io`
- `alb.ingress.cpaas.io`
- `nginx.ingress.kubernetes.io`

Это позволяет решить проблему совместимости с ingress-nginx и задать конфигурацию аутентификации для конкретного пути Ingress.

### Auth-Enable

```yaml
alb.ingress.cpaas.io/auth-enable: "false"
```

Новая аннотация, добавленная ALB, используется для указания, следует ли включать функциональность аутентификации для Ingress.

## Другие функции, связанные с аутентификацией Ingress-Nginx

### Global-Auth

В ingress-nginx вы можете установить глобальную аутентификацию через ConfigMap. Это эквивалентно настройке аутентификации для всех Ingress. В ALB вы можете настроить аутентификацию на ALB2 и FT CR. Правила под ними будут наследовать эти конфигурации.

### No-Auth-Locations

В ALB вы можете отключить функцию аутентификации для этого Ingress, настроив аннотацию: `alb.ingress.cpaas.io/auth-enable: "false"` на Ingress.

## Примечание: Несовместимые части с Ingress-Nginx

1. Не поддерживает auth-keepalive
2. Не поддерживает auth-snippet
3. Не поддерживает auth-cache
4. Не поддерживает auth-tls
5. Basic-auth поддерживает только basic, не поддерживает digest
6. Basic-auth basic поддерживает только алгоритм apr1, не поддерживает bcrypt sha256 и др.

## Устранение неполадок

1. Проверьте логи Nginx контейнера ALB pod
2. Проверьте заголовок `X-ALB-ERR-REASON` в ответе

# 介绍

OAM 应用实现了对容器和 Kubernetes 资源的更高层整合和抽象，可帮助您更专注于应用的开发与运维。

## 什么是 OAM 

开放应用模型（Open Application Model，OAM）是用于定义云原生应用的模型，专注于描述应用的标准。与容器或 Kubernetes 编排逻辑不同，OAM 更加关注 **应用** 本身。

- 开放（Open）：支持异构的平台、容器运行时、调度系统、云服务商、硬件配置等，独立于底层基础设施。

- 应用（Application）：云原生应用，能够复用多种基础设施。

- 模型（Model）：定义标准，使其与底层平台无关。

## 为什么使用 OAM

云原生应用是多个相互关联但离散的组件集合，这些组件通过配置在适合的基础设施中实例化，以实现统一的功能。然而，各个组件和功能之间紧密相连，使得开发、交付和管理的过程变得相当复杂。

OAM 为上层应用提供了良好的抽象系统，即使能力的实现层被完全替换，也不会影响上层应用，真正实现与基础设施无关。此外，OAM 模型旨在解决的核心问题不是如何编写程序，而是如何组合分布式应用，从而通过 CUE 实现交付过程的轻松扩展、复用或重写。

不再需要难以维护的脚本和复杂的衔接代码，最终用户无需关心任何基础设施细节，他们只需专注于定义和部署应用。一旦编排完成，应用可以在任何地方运行。

## OAM 与其他应用对比

### OAM 应用与原生应用

OAM 应用是一个基于云原生技术栈构建的现代应用交付系统。它利用 OAM 模型和 Kubernetes 作为控制平面，以解决一个长期存在的挑战：如何简化应用交付。因此，OAM 应用比原生应用更具用户友好性，交付过程更加简便。

### OAM 应用与 GitOps 应用

OAM 应用可以基于 GitOps 流程，并在其基础上增强跨云和跨环境的能力：

- OAM 应用提供用户友好且可编程的工作流，能够集成现有的交付工具。
- OAM 应用支持跨环境交付，允许描述多集群的差异化配置并统一监控状态。

### OAM 应用与模板应用

Helm 是 Kubernetes 的包管理器，它能够提供一组 YAML 文件以 Chart 为单位进行打包、安装和升级。

作为应用交付系统，OAM 可以部署多种类型的制品，如 Kustomize、YAML 等，当然也包括 Charts。Helm 方便地将 Charts 交付到单个集群，而 OAM 则可以帮助将 Charts 交付到多个集群，同时解决配置漂移的问题。

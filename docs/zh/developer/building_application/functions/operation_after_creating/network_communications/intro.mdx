# 不同 ingress 方法之间的比较

ACP 平台支持 Kubernetes 生态系统中的多种 ingress 流量规格。本文将对它们（Service、Ingress、Gateway API 和 ALB 规则）进行比较，以帮助用户作出正确的选择。

## 对于 L4（TCP/UDP）流量

LoadBalancer 类型的 Service、Gateway API 和 ALB 规则都可以将 L4 流量暴露到外部。在这里，我们推荐使用 LoadBalancer 类型的 Service 方法。Gateway API 和 ALB 规则都是由 ALB 实现的，ALB 是用户空间代理，在处理 L4 流量时，其性能相比 LoadBalancer 类型的 Service 显著下降。

## 对于 L7（HTTP/HTTPS）流量

虽然 Ingress、Gateway API 和 ALB 规则都可以将 L7 流量暴露到外部，但它们在能力和隔离模型上有所不同。

### Ingress

Ingress 是 Kubernetes 社区采用的标准规格，推荐作为默认使用。Ingress 由平台管理员管理的 ALB 实例处理。

### Gateway API

Gateway API 提供了更灵活的隔离模式，但其成熟度不及 Ingress。通过使用 Gateway API，开发人员可以创建自己隔离的 ALB 实例来处理 Gateway API 规则。因此，如果需要将 ALB 实例的创建和管理委托给开发人员，则需选择使用 Gateway API。

### ALB 规则

ALB 规则（UI 中的负载均衡器）提供了最灵活的流量匹配规则和最多的功能。实际上，Ingress 和 Gateway API 都是通过将它们翻译为 ALB 规则来实现的。但是，ALB 规则比 Ingress 和 Gateway API 更复杂，并且不是社区标准 API。因此，我们建议仅在 Ingress 和 Gateway API 无法满足需求时使用它。

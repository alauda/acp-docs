# 介绍

## 容器平台首次使用流程

若您首次使用 Container Platform，可参考 [使用镜像快速部署一个简单应用](./quick_start/quick_start_acp.mdx) 文档，了解基本的使用方式。

## 应用类型

在平台的 **Container Platform > 应用管理** 中，提供了如下可创建的应用类型：

* **原生应用**：由一个或多个关联的计算组件（Workloads）、内部路由（Services）等 Kubernetes 原生资源构成的整体业务应用。支持通过 UI 编辑、YAML 编排，以及模板等方式创建，可运行在开发、测试或生产环境中。可以通过如下方式创建不同类型的原生应用。
    * **通过镜像创建**：使用已有的容器镜像快速创建应用。
    * **通过 Helm Chart 创建**：使用 Helm Chart 包创建应用。
    * **通过 YAML 创建**：使用 YAML 配置文件创建应用。
    * **通过代码创建**：使用源代码创建应用。
* **组件应用**：基于应用组件（Operator backed）您可以快速部署一个组件应用，并借助 Operator 的能力自动化管理应用的整个生命周期。
* **OAM 应用**：用于定义云原生应用的模型。相比容器或 Kubernetes 编排逻辑，OAM 更关注"应用"本身。基于 OAM，应用的通用能力都被封装成高层次的接口供使用，并贯穿应用的部署、开发和运维全过程。

## 工作负载类型

平台除了可以创建原生应用、组件应用等，还可以直接在 **Container Platform > 计算组件** 中创建工作负载：

* **Deployment**：是最常用的工作负载控制器，用于部署无状态应用。它可以确保指定数量的 Pod 副本在集群中运行，支持滚动更新和回滚，适用于 Web 服务、API 服务等无状态应用场景。

* **DaemonSet**：确保集群中的每个节点（或特定节点）都运行一个 Pod 副本。当节点加入集群时，Pod 会被自动创建；当节点从集群中移除时，这些 Pod 也会被回收。适用于日志收集、监控采集等需要在每个节点上运行的场景。

* **StatefulSet**：用于管理有状态应用的工作负载控制器。它为每个 Pod 维护了一个固定的标识，并提供稳定的存储和网络标识，即使 Pod 重新调度也会保持不变。适用于数据库、分布式缓存等有状态应用。

* **Job**：用于运行一次性任务的工作负载。Job 会创建一个或多个 Pod 并确保指定数量的 Pod 成功完成任务后结束。适用于批处理、数据迁移等一次性任务场景。

* **CronJob**：用于管理基于时间计划运行的 Job。您可以设置任务执行的时间表达式，系统会按照预定的时间自动创建和运行 Job。适用于数据备份、报表生成、定期清理等周期性任务场景。

除了通过平台表单页面创建上述计算组件外，平台还支持通过 CLI 工具创建 Pod 和 Container：

* **Pod**：是 Kubernetes 中最小的可部署单元，一个 Pod 中可以包含一个或多个容器，这些容器共享存储、网络和配置声明。Pod 通常由控制器（如 Deployment）管理。
* **Container**：容器是一个标准的软件单元，它打包了代码和所有依赖项，以便应用程序可以在不同的计算环境中快速可靠地运行。容器在 Pod 内运行，共享 Pod 的资源。

# 介绍

如果您首次使用容器平台，您可以参考文档 [使用镜像快速部署一个简单应用](./quick_start/quick_start_app.mdx) 以了解基本的使用方式。

## 应用类型

在平台的 **Container Platform > 应用管理** 中，可以创建以下类型的应用：

- **应用**：由一个或多个关联的计算组件（Workloads）、内部路由（Services）和其他原生 Kubernetes 资源组成的完整业务应用。支持通过 UI 编辑、YAML 编排和模板等方式创建，并可以在开发、测试或生产环境中运行。不同类型的原生应用可以通过以下方式创建：
  - **通过镜像创建**：使用现有的容器镜像快速创建应用。
  - **通过目录创建**：使用 Helm Chart 包创建应用。
  - **通过 YAML 创建**：使用 YAML 配置文件创建应用。
  - **通过代码创建**：使用源代码创建应用。
- **组件应用**：基于应用组件（Operator backed），您可以快速部署一个组件应用，并利用 Operators 的能力自动化管理应用的整个生命周期。
- **OAM 应用**：用于定义云原生应用的模型。与容器或 Kubernetes 编排逻辑相比，OAM 更加关注“应用”本身。基于 OAM，应用的常见功能被封装成高层次的接口供使用，贯穿应用的部署、开发和运维全过程。

## 工作负载类型

除了可以创建原生应用和组件应用外，还可以直接在 **Container Platform > 计算组件** 中创建工作负载：

- **Deployment**：用于部署无状态应用的最常用工作负载控制器。它可以确保指定数量的 Pod 副本在集群中运行，支持滚动更新和回滚，适用于 Web 服务和 API 服务等无状态应用场景。

- **DaemonSet**：确保集群中的每个节点（或特定节点）都运行一个 Pod 副本。当节点加入集群时，Pod 会被自动创建；当节点从集群中移除时，这些 Pods 也会被回收。适合需要在每个节点上运行的日志、监控等场景。

- **StatefulSet**：用于管理有状态应用的工作负载控制器。它为每个 Pod 维护固定的身份，并提供稳定的存储和网络身份，即使 Pod 重新调度也不会改变。适合有状态的数据库和分布式缓存等应用。

- **Job**：用于运行一次性任务的工作负载。Job 会创建一个或多个 Pods，并确保指定数量的 Pods 成功完成任务后终止。适合批处理、数据迁移等一次性任务场景。

- **CronJob**：用于管理按时间计划运行的 Jobs。您可以设置任务执行的时间表达式，系统将在预定的时间自动创建并运行 Job。适合数据备份、报表生成和定期清理等周期性任务。

除了通过平台的表单页面创建上述计算组件外，平台也支持通过 CLI 工具创建 Pods 和 Containers：

- **Pod**：Kubernetes 中最小的可部署单元，一个 Pod 可以包含一个或多个容器，这些容器共享存储、网络和配置声明。Pods 通常由控制器（如 Deployments）管理。
- **Container**：一个标准的软件单元，打包了代码和所有依赖项，使得应用能够在不同的计算环境中快速、可靠地运行。Containers 在 Pods 内部运行，并共享 Pods 的资源。

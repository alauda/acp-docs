# 介绍

OAM 应用实现了对容器和 kubernetes 资源的更高层整合和抽象，可帮助您更专注于应用的开发与运维。


## 什么是 OAM 

开放应用模型（Open Application Model，OAM）是用于定义云原生应用的模型，是一个专注于描述应用的标准规范。相比容器或 Kubernetes 编排逻辑，OAM 更关注 **应用** 本身。

* 开放（Open）：支持异构的平台、容器运行时、调度系统、云供应商、硬件配置等，与底层无关。

* 应用（Application）：云原生应用，可以复用各种基础设施。

* 模型（Model）：定义标准，以使其与底层平台无关。


## 为什么使用 OAM

云原生应用程序本是相互关联但离散的组件集合，它们与配置相结合，并在合适的基础设施中实例化，共同实现统一的功能，但是每个组件、每个功能之间紧密联系相互制约，导致开发、交付、管理等流程非常复杂。

OAM 为上层应用提供了良好的抽象体系，能力的实现层即使被完全替换也不会影响上层应用，真正做到与基础设施无关，且 OAM 模型寻求解决的核心问题不在于如何编写程序，而在于如何组合分布式应用程序，通过 CUE 轻松扩展、复用或重新编写你的交付过程。

再也没有难以维护的脚本和复杂的衔接代码，最终用户无需关注任何基础设施细节，只需专注于定义和部署应用，一次编排、随处运行。

## OAM 与其他应用对比

### OAM 应用与原生应用

OAM 应用是一个基于云原生技术栈构建的现代应用交付系统。它利用了 OAM 模型和 Kubernetes 作为控制平面来解决一个旷日已久的难题：如何让应用交付变得更加轻松。因此 OAM 应用比原生应用更易使用、交付更便捷。

### OAM 应用与 GitOps 应用

OAM 应用可以基于 GitOps 流程，并在此之上增加跨云、跨环境的能力：

* OAM 应用具有用户友好且可编程的工作流，可以集成现有的交付工具。
* OAM 应用可以为提供跨环境交付能力，可以在应用中描述多集群的差异化配置并统一的查看状态。


### OAM 应用与模板应用

Helm 是 Kubernetes 的包管理器，它能够以 Chart 为一个单元，提供打包、安装和升级的一组 YAML 文件的能力。

OAM 作为一个应用交付系统，可以部署各种制品类型：Kustomize、YAML 等，当然也包括 Chart。其中，Helm 可以便捷的把 Chart 交付到一个集群，而 OAM 可以帮你把 Chart 交付到多个集群，同时解决了配置漂移的问题。

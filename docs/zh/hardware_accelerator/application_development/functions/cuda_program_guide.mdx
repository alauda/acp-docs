# CUDA 驱动程序与运行时兼容性

## 分层架构和核心概念

![cuda rt](../assets/cuda-arti.png "Cuda driver runtime layver")

### 1. CUDA 运行时 API 层

#### 技术定位

1. **功能范围**：为开发人员提供高层次的抽象接口，封装核心 GPU 操作（内存分配、流管理、内核启动等）
2. **版本绑定**：由构建时所使用的 CUDA 工具包版本决定（例如，CUDA 12.0.1）

#### 版本检测方法

```bash
# Python 环境检测（推荐优先使用方法）
pip list | grep cuda
conda list | grep cuda
# 示例输出：cu121  # cu121 表示 CUDA 12.1 环境

# 系统级运行时库检测
find / -name "libcudart*"
# cudart 表示 CUDA 运行时
# 示例输出：
/usr/local/cuda-12.4/targets/x86_64-linux/lib/libcudart.so.12
/usr/local/cuda-12.4/targets/x86_64-linux/lib/libcudart.so.12.4.127
表示 CUDA 12.4
```

如果找到多个库版本，您应检查您的程序使用了哪个版本，参考 PATH、LD_LIBRARY_PATH 或其他程序设置。

```bash
env | grep PATH
# 示例输出：
LIBRARY_PATH=/usr/local/cuda/lib64/stubs
LD_LIBRARY_PATH=/usr/local/nvidia/lib:/usr/local/nvidia/lib64
PATH=/go/bin:/usr/local/go/bin:/usr/local/nvidia/bin:/usr/local/cuda/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
# 这意味着您的 CUDA 程序使用了按库路径顺序排列的第一个库
```

### 2. CUDA 驱动 API 层

#### 技术定位

1. **功能范围**：直接与 GPU 硬件交互的低层接口，处理指令翻译和硬件资源调度
2. **版本绑定**：由 NVIDIA 驱动版本决定，遵循 SemVer 规范

## 版本兼容矩阵与约束

### 物理 GPU 部署 - 核心兼容性原则

参照 NVIDIA 的官方声明，**基本约束**如下：

1. **驱动版本必须始终 ≥ 运行时版本**
2. NVIDIA 官方保证 **向后兼容 1 个主要版本**（例如，CUDA 驱动 12.x 支持运行时 11.x）
3. 跨两个主要版本的兼容性（例如，驱动 12.x 与运行时 10.x）既未正式支持也不推荐

在部署 CUDA 程序时，请遵循基本约束。

#### 正式规则

```diff
+ 强制：驱动版本 ≥ 运行时版本
+ 推荐：驱动主版本 - 运行时主版本 ≤ 1
- 阻止：驱动版本 < 运行时版本 → 可能触发 CUDA_ERROR_UNKNOWN(999)
- 不稳定：驱动主版本 - 运行时主版本 > 1 → 应用可能出现故障
```

### 虚拟化场景增强（HAMI/GPU-Manager）

在使用类似 GPU-Manager 或 HAMI 的虚拟 GPU 解决方案时，**除了基本约束**，您还必须遵循 **附加约束**：

#### 版本要求

```text
1. 虚拟 GPU 解决方案基础版本 ≥ 运行时版本
2. 运行时主版本 = 驱动主版本 = 基础主版本
```

**GPU-Manager 特别说明**：
我们实现了部分跨 1 个主要版本的兼容性（例如，基础版本 12.4 支持 vLLM 11.8）。但是，这需要针对每个应用程序的钩子调整，并且必须逐个案例分析。

## 部署最佳实践

### 推荐策略

• 在新的 GPU 集群规划中采用更新的 CUDA 版本（例如，CUDA 12.x）作为驱动和运行时

### 旧系统替代方案

#### 1. 物理 GPU 调度或 GPU-Manager 整卡分配

整卡调度提供与物理 GPU 访问相当的原生兼容性。在设置 tencent.com/vcuda-core 为 100 的正整数时，GPU-Manager 可以使用整卡模式，示例值为 100、200、300。

```yaml
resources:
  limits:
    tencent.com/vcuda-core: "100"
```

#### 2. 节点标签策略

根据支持的驱动程序 CUDA 版本给节点打标签：

```yaml
node_labels:
  cuda-major-version: "12"
  cuda-minor-version: "4"
```

这意味着您的节点为 CUDA 12.4。

在部署中配置调度亲和性：
您可以根据程序的 CUDA 运行时需求设置 cuda-major-version 和 cuda-minor-version。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cuda-app
spec:
  template:
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: cuda-major-version
                operator: In
                values: ["12"]
              - key: cuda-minor-version
                operator: Gt
                values: ["2"]
```

#### 3. 运行时版本升级

旧版 CUDA 运行时可能存在安全漏洞（CVE）且缺乏对新 GPU 功能的支持。优先升级到 CUDA 12.x。

NVIDIA 推荐同时升级两者。
![upgrade both](../assets/upgradeboth.png "upgrade both")

## 故障排除手册

### 常见错误代码

| 错误代码                             | 描述                                           | 推荐操作                               |
| -------------------------------------- | ------------------------------------------------ | ------------------------------------------------ |
| CUDA_ERROR_INVALID_IMAGE            | 驱动程序-运行时不兼容                        | 使驱动版本与容器 CUDA 版本对齐 |
| CUDA_ERROR_ILLEGAL_ADDRESS          | 虚拟内存违规（常见于版本不匹配） | 验证运行时与基础约束            |
| CUDA_ERROR_UNSUPPORTED_PTX_VERSION | PTX 指令集不匹配                          | 重新编译时明确指定 -arch=sm_xx |
